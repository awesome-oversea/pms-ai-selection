# 本地排障、重置与恢复手册

> 适用目录：`D:\Project\fms`
>
> 目标：沉淀 `N0-05` 本地交付基线，确保项目在“启动失败、依赖异常、数据脏状态、缓存失效、样例工件被污染”场景下，有统一的排障、重置、恢复和演练口径。

---

## 1. 适用范围

本手册覆盖以下本地优先场景：

- 启动入口、依赖探活和配置检查
- 本地数据状态重置
- 本地 ERP 样例工件恢复
- 本地反馈回流重建
- 本地恢复演练与证据落盘

不直接覆盖生产环境故障切换。生产向 backup/recovery 基线可继续复用：

```bash
rtk python scripts/ha_backup_recovery_check.py
```

---

## 2. 统一命令入口

### 2.1 运行态检查

```bash
rtk python scripts/local_runtime_manager.py summary
rtk python scripts/local_runtime_manager.py check --probes
```

适用：

- 判断当前 `.env`、运行模式和依赖计划是否一致
- 判断 DB / Redis / Qdrant / Search / Ollama / Dify 探活状态

### 2.2 恢复管理

```bash
rtk python scripts/local_recovery_manager.py status
rtk python scripts/local_recovery_manager.py reset --include-report-state
rtk python scripts/local_recovery_manager.py restore
rtk python scripts/local_recovery_manager.py drill
```

说明：

- `status`
  查看本地恢复关键目标是否存在，如 `.env`、本地数据文件、`artifacts/erp_local`、`artifacts/report_center/state.json`
- `reset`
  删除本地数据状态，默认重置 `data/local_feature_store.db` 和 `data/local_knowledge.db`
- `restore`
  将 `artifacts/erp_local` 基线重新恢复到指定工作区
- `drill`
  执行一次完整的“污染状态 → 重置 → 恢复 → 回流重建”演练，并输出标准工件

## 3. 常见故障与处理顺序

### 3.1 启动失败或运行模式混乱

先执行：

```bash
rtk python scripts/local_runtime_manager.py summary
rtk python scripts/local_runtime_manager.py check --probes
```

处理重点：

- `.env` 是否存在且 `LOCAL_RUNTIME_*` 配置正确
- `SEARCH_BACKEND` 与 `SEARCH_ENDPOINT` 是否匹配
- `SERVICE_MODE_*` 与 `LOCAL_RUNTIME_SCENARIO_MODE` 是否冲突

### 3.2 本地样例数据脏状态

表现：

- 本地回流结果异常
- 报告状态文件被污染
- 本地知识/特征库残留旧状态

建议顺序：

```bash
rtk python scripts/local_recovery_manager.py status
rtk python scripts/local_recovery_manager.py reset --include-report-state
rtk python scripts/local_recovery_manager.py restore
```

### 3.3 本地知识回流或特征库异常

表现：

- `data/local_feature_store.db` 数据异常
- `data/local_knowledge.db` 存在旧 review case 干扰
- 本地反馈回流结果与样例不一致

建议顺序：

```bash
rtk python scripts/local_recovery_manager.py reset
rtk python scripts/local_recovery_manager.py restore
rtk python scripts/local_recovery_manager.py drill
```

`drill` 会在隔离工作区中执行一次完整重建，避免直接污染共享目录。

### 3.4 本地 Qdrant 运行态异常

如只需清理本地嵌入式 Qdrant 数据，可执行：

```bash
rtk python scripts/local_recovery_manager.py reset --include-qdrant-local
```

该操作会删除：

- `artifacts/runtime/qdrant_local`

执行前应确认当前不是要保留已有本地向量数据的场景。

## 4. 标准恢复流程

### 场景 A：恢复本地 ERP 样例工件

```bash
rtk python scripts/local_recovery_manager.py restore
```

预期：

- `artifacts/erp_local/oms/orders.json` 存在
- `artifacts/erp_local/crm/feedback.json` 存在

### 场景 B：重置本地数据状态

```bash
rtk python scripts/local_recovery_manager.py reset --include-report-state
```

预期被清理：

- `data/local_feature_store.db`
- `data/local_knowledge.db`
- `artifacts/report_center/state.json`

### 场景 C：执行完整恢复演练

```bash
rtk python scripts/local_recovery_manager.py drill
```

输出目录：

- `artifacts/local_recovery/<run_id>/summary.json`
- `artifacts/local_recovery/<run_id>/restore_report.json`
- `artifacts/local_recovery/<run_id>/reset_report.json`
- `artifacts/local_recovery/<run_id>/rebuild_report.json`

通过标准：

- 基线工件可恢复
- 本地数据状态可清理
- 回流重建后重新生成 `data/local_feature_store.db` 与 `data/local_knowledge.db`
- `summary.json` 中 `accepted=true`

## 6. 与 N0/N1 的关系

- `N0-01` ~ `N0-04` 负责把本地启动、配置、样例和验收入口做成统一基线
- `N0-05` 负责在这套基线之上补齐“怎么恢复、怎么清理、怎么重建、怎么留证据”
- `N1` 进入业务闭环前，应先确保本手册和恢复演练可用，否则多角色联调会反复卡在环境状态不一致
