# 依赖栈与组件说明

本文档只描述当前仍然有效的本地依赖栈。

## 1. 依赖栈总表

| 栈 | 文件 | 主要服务 | 说明 |
|---|---|---|---|
| 基础数据栈 | `docker-compose.yml` | `postgres` `redis` `qdrant` | 默认本地基础运行栈；`app` / `crawler-worker` 为显式 profile 服务 |
| Kafka / CDC 栈 | `docker-compose.local-kafka.yml` | `zookeeper` `kafka` `kafka-init` `kafka-connect` `debezium-init` | 唯一 Kafka 本地栈 |
| 网关 / 搜索 / 图谱栈 | `docker-compose.wsl-local.yml` | `kong-gateway` `opensearch` `neo4j` | local-real 推荐附加栈 |
| 平台栈 | `docker-compose.wsl-platform.yml` | `redis-master` `redis-replica-*` `redis-sentinel-*` `keycloak` `flink-*` | OIDC / Sentinel / Flink |
| PostgreSQL HA 栈 | `docker-compose.wsl-postgres-ha.yml` | `pg-primary` `pg-standby-*` `pgpool` | 主从与连接池 |
| Qdrant Cluster 栈 | `docker-compose.wsl-qdrant-cluster.yml` | `qdrant-node-*` | 本地高可用演练 |

## 2. 基础数据栈

文件：`D:/Project/fms/docker-compose.yml`

当前只包含：

- `postgres`
- `redis`
- `qdrant`
- `crawler-worker`（显式 `worker-jobs` profile）
- `app`（显式 `app-image` profile）

标准启动：

```bash
rtk docker compose -f D:/Project/fms/docker-compose.yml up -d postgres redis qdrant
```

说明：

- backend 默认走宿主机 Python 进程；如需显式启用镜像模式，再执行 `rtk docker compose --profile app-image -f D:/Project/fms/docker-compose.yml up -d --build app`
- `app` 镜像模式仍映射宿主机 `18000:8000`
- `8000` 留给 Kong
- 这里不再启动 Kafka 相关服务

## 3. Kafka / CDC 栈

文件：`D:/Project/fms/docker-compose.local-kafka.yml`

当前只保留这一套 Kafka 本地栈：

- `zookeeper`
- `kafka`
- `kafka-init`
- `kafka-connect`
- `debezium-init`

标准启动：

```bash
rtk python D:/Project/fms/scripts/start_local_services.py --with-kafka
```

底层高级入口：

```bash
rtk python D:/Project/fms/scripts/bootstrap_local_kafka_debezium.py --startup-only
```

完整验收入口：

```bash
rtk python D:/Project/fms/scripts/bootstrap_local_kafka_debezium.py
```

### 3.1 为什么业务配置不需要改

这套 Kafka 栈会挂到共享 `pms-network`，并暴露以下别名：

- `kafka`
- `pms-local-kafka`
- `kafka-connect`
- `pms-local-kafka-connect`
- `zookeeper`
- `pms-local-zookeeper`

因此：

- `app` 仍然可以继续使用 `kafka:29092`
- 需要访问 Kafka Connect 的服务仍然可以继续使用 `kafka-connect:8083`
- 不需要为了这次收敛去修改现有业务配置、环境变量或代码路径

### 3.2 数据与状态

Kafka 栈当前持久化到命名卷：

- `local_zookeeper_data`
- `local_zookeeper_log`
- `local_kafka_data`

说明：

- 这保证了 Kafka 和 ZooKeeper 的状态不随容器重建而丢失
- 不要对这套栈执行 `docker compose down -v`

## 4. 网关 / 搜索 / 图谱栈

文件：`D:/Project/fms/docker-compose.wsl-local.yml`

主要服务：

- `kong-database`
- `kong-migrations`
- `kong-gateway`
- `opensearch`
- `neo4j`

说明：

- Kong upstream 已统一指向 `host.docker.internal:18000`
- 这同时兼容宿主机 Python backend 和显式启用的 `app` 镜像模式

## 5. 平台栈

文件：`D:/Project/fms/docker-compose.wsl-platform.yml`

主要服务：

- Redis 主从与 Sentinel
- Keycloak
- Flink JobManager / TaskManager

标准启动：

```bash
rtk python D:/Project/fms/scripts/start_local_services.py --with-platform
```

## 6. 配置边界

当前应坚持下面几条：

- 不要重新把 Kafka / Zookeeper / Kafka Connect 加回 `docker-compose.yml`
- 不要再维护第二套 Kafka 容器命名
- 不要让文档同时给出两套 Kafka 启动方式
- 业务容器继续按原语义使用 `kafka:29092`
