# 第一梯队可执行开发任务清单

## 1. 说明

本文基于 `docs/企业级进阶任务清单.md` 的“第一梯队”任务进行进一步拆解，目标是把高优先级方向转换为可以进入迭代排期、分工和验收的执行清单。  
范围聚焦以下 4 项：

1. 建立正式前端工程与 BFF 层  
2. 引入真实 API 网关治理  
3. 建立 Elasticsearch/OpenSearch 关键词检索层  
4. 落地一个真实 ERP/业务系统集成样板  

另补充一个横向保障项：**蓝图能力与当前能力双轨文档机制**，作为第一梯队期间持续维护项。

---

## 2. 总体推进顺序与依赖

### 推荐顺序
1. 前端工程与 BFF 层
2. API 网关治理
3. ES/OpenSearch 关键词检索层
4. ERP/业务系统集成样板

### 关键依赖关系
- 前端/BFF 是网关治理和开放接口治理的直接消费方。
- 网关治理完成后，开放接口、BFF、内部接口边界才可稳定固化。
- ES/OpenSearch 检索层依赖现有知识库、租户隔离、混合检索基础。
- ERP 样板集成依赖统一接口边界、审计、租户、任务编排与配置中心。

### 建议节奏
- **里程碑 M1（2~3 周）**：前端/BFF 基线 + 文档机制
- **里程碑 M2（2 周）**：网关治理最小闭环
- **里程碑 M3（2~3 周）**：ES/OpenSearch 检索闭环
- **里程碑 M4（3~4 周）**：ERP 样板集成闭环

---

## 3. T1 建立正式前端工程与 BFF 层

- **优先级**：P0
- **现状**：当前仅有 `web/templates` 和 `src/web/routes.py`，属于服务端模板页面，不具备企业级前端工程能力。
- **目标**：建立独立前端工程与 BFF 层，形成工作台级 UI、会话鉴权、前后端契约分层。
- **负责人角色**：前端负责人、BFF/后端负责人、产品/交互负责人、测试负责人
- **预计工期**：10~15 个工作日

### 里程碑
- **M1.1**：建立前端工程脚手架（Next.js/Vite + React）
- **M1.2**：建立 BFF 路由分层与接口代理
- **M1.3**：完成至少 1 个工作台（推荐：选品任务工作台）
- **M1.4**：完成登录态、菜单、错误处理、基础埋点

### 依赖关系
- 依赖当前统一 API 契约、RBAC、租户上下文、审计日志基线
- 后续依赖网关治理提供更明确的访问边界

### 风险点
- 页面与现有 API 契约不完全匹配
- BFF 与后端职责边界模糊
- 前后端并行开发时可能出现契约漂移

### 实施步骤
1. 创建独立前端目录与工程化基础
2. 定义 BFF 路由命名规范、鉴权透传规范、错误映射规范
3. 将现有模板页面功能迁移到前端页面
4. 增加基础状态管理、接口 SDK 和环境配置
5. 逐步淘汰模板渲染页，仅保留兼容入口

### 测试用例
- 登录后可进入工作台并获取当前租户数据
- 未登录访问被拦截
- BFF 对 401/403/429/503 错误映射稳定
- 至少 1 个任务创建/查询页面可真实联通

### 验收标准
- 存在独立前端工程
- 存在 BFF 路由层
- 至少 1 个正式工作台真实可运行
- 页面不再依赖后端模板拼装核心业务逻辑

### 预计影响范围
- `web/`
- `src/web/`
- `src/api/`
- 鉴权与接口契约

## 4. T2 引入真实 API 网关治理

- **优先级**：P0
- **现状**：当前只有应用内中间件、限流和接口治理元数据，没有基础设施级网关。
- **目标**：形成真实网关层，统一承接鉴权、限流、版本、开放接口策略。
- **负责人角色**：后端架构负责人、DevOps/平台负责人、安全负责人、测试负责人
- **预计工期**：8~12 个工作日

### 里程碑
- **M2.1**：选型并确定网关方案（Kong/APISIX/云网关）
- **M2.2**：定义 API 分组：internal / bff / openapi
- **M2.3**：完成 1 组真实路由接入与策略生效
- **M2.4**：完成限流、版本、消费方策略验证

### 依赖关系
- 依赖 T1 前端/BFF 分层
- 依赖现有接口治理元数据和统一 API 契约

### 风险点
- 双层鉴权（应用内 + 网关）职责冲突
- 开放接口策略和内部接口边界混淆
- 本地/测试环境网关调试成本上升

### 实施步骤
1. 选定网关产品与部署模式
2. 将当前 `/api/v1` 接口按 internal/bff/openapi 分层标注
3. 配置真实路由、认证、配额、限流和版本策略
4. 为开放接口建立消费方标识与访问控制
5. 将治理规则与 OpenAPI 元数据对齐

### 测试用例
- BFF 接口只能通过 BFF 路由暴露
- Open API 路径存在限流和版本规则
- 非授权调用方无法访问开放接口
- 网关 429/401/403 行为可重复验证

### 验收标准
- 至少 1 套真实网关配置生效
- internal / bff / openapi 边界清晰
- 限流、版本、开放接口访问规则可验证

### 预计影响范围
- 部署架构
- OpenAPI
- 鉴权与限流
- API 暴露边界

## 5. T3 建立 Elasticsearch/OpenSearch 关键词检索层

- **优先级**：P1
- **现状**：当前为 Qdrant + 本地 BM25 融合，缺正式关键词召回基础设施。
- **目标**：建立 ES/OpenSearch 关键词索引与召回层，形成稳定混合检索。
- **负责人角色**：后端负责人、搜索/算法负责人、平台负责人、测试负责人
- **预计工期**：10~15 个工作日

### 里程碑
- **M3.1**：建立文档/分块索引结构
- **M3.2**：实现 tenant/kb/doc 过滤查询
- **M3.3**：接入与 Qdrant 的融合策略
- **M3.4**：完成效果验证样例与运营指标

### 依赖关系
- 依赖已完成的 T7.1/T7.2/T7.3
- 后续可被 T7.4 质量评估持续消费

### 风险点
- ES 与 Qdrant 索引一致性问题
- 多租户过滤错误导致数据串租户
- 检索效果提升不明显，增加运维成本

### 实施步骤
1. 设计 ES/OpenSearch index mapping 与字段口径
2. 建立写入/更新/删除同步逻辑
3. 接入混合检索融合层
4. 输出召回质量和索引状态指标
5. 对关键查询样例做效果对比验证

### 测试用例
- tenant / kb / doc 过滤有效
- 文档更新后 ES 索引同步更新
- 混合检索结果同时包含向量与关键词召回证据
- 至少 1 组查询效果优于当前基线

### 验收标准
- 向量与 ES 关键词召回同时可用
- 过滤条件生效
- 有明确的效果验证样例和指标

### 预计影响范围
- `knowledge_service`
- 检索链路
- 部署资源
- 观测体系

## 6. T4 落地一个真实 ERP/业务系统集成样板

- **优先级**：P1
- **现状**：当前只有主数据/配置/同步方向定义，没有真实双向系统集成闭环。
- **目标**：选定 1 个 ERP/业务系统，建立真实数据同步样板。
- **负责人角色**：后端负责人、集成负责人、业务分析负责人、测试负责人、运维负责人
- **预计工期**：15~20 个工作日

### 里程碑
- **M4.1**：确定目标系统与集成范围
- **M4.2**：定义主权字段、同步方向、冲突处理
- **M4.3**：实现 1 条入站 + 1 条出站同步链路
- **M4.4**：完成失败重试、审计、状态查询

### 依赖关系
- 依赖统一数据域字典（T7.5）
- 依赖配置中心（T7.6）
- 依赖网关治理（T8.3）和审计能力

### 风险点
- 外部系统 API 不稳定或权限限制
- 主数据口径不清导致字段冲突
- 重试与幂等处理不当导致脏数据

### 实施步骤
1. 选定样板系统（建议先从 OMS/SCM 二选一）
2. 定义集成对象、字段映射、同步方向与 SLA
3. 实现连接配置、拉取/推送、审计日志与失败重试
4. 补充人工重试与状态查询入口
5. 输出集成运行说明和异常处理手册

### 测试用例
- 配置正确时可完成至少 1 次真实同步
- 重复同步不产生重复数据
- 外部接口失败时可重试并记录审计
- 同步状态可查询、失败可人工处理

### 验收标准
- 至少 1 个系统完成真实双向样板闭环
- 有审计、重试、状态查询与失败处理机制
- 字段主权和同步方向明确可追踪

### 预计影响范围
- `models`
- `services`
- `integration`
- 审计与任务系统

## 7. 横向保障项：蓝图能力与当前能力双轨文档机制

- **优先级**：P0（持续项）
- **现状**：设计蓝图与当前交付能力容易被混淆。
- **目标**：保证管理层、研发、实施团队对“已交付能力”和“目标态能力”口径一致。
- **负责人角色**：架构负责人、项目经理、研发负责人
- **预计工期**：持续维护；首轮整理 2~3 个工作日

### 里程碑
- **D1**：维护当前能力边界文档
- **D2**：维护目标态蓝图文档
- **D3**：维护进阶任务清单与优先级

### 风险点
- 汇报中夸大当前能力
- 评审和实施时误把目标态当成交付态

### 实施步骤
1. 固定三类文档入口
2. 每阶段验收后同步更新
3. 评审/汇报统一引用文档版本

### 验收标准
- 三类文档长期维护
- 任一重大能力变更能同步反映在三类文档中

---

## 8. 第一梯队最终建议

若资源有限，建议按以下顺序切迭代：

- **迭代 1**：前端工程 + BFF 基线 + 双轨文档机制
- **迭代 2**：真实 API 网关治理
- **迭代 3**：ES/OpenSearch 关键词检索层
- **迭代 4**：ERP/业务系统集成样板

推荐原则：
- 先补“用户可见与边界治理”
- 再补“检索基础设施”
- 最后补“真实外部集成样板”

这样能在控制风险的同时，持续提升系统从“企业级单体平台”向“企业级业务中台”演进的可执行性。
