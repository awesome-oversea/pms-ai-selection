# 第二梯队可执行开发任务清单

## 1. 说明

本文基于 `docs/企业级进阶任务清单.md` 的“第二梯队”任务进一步拆解，目标是把平台拆分阶段的工作转化为可排期、可分工、可验收的开发清单。  
本梯队聚焦以下 2 项：

1. 拆分 RAG / LLM 为独立服务  
2. 建立 Kafka/CDC 驱动的数据同步样板

本梯队的核心目标不是“大规模微服务化”，而是：**在不打断当前单体交付能力的前提下，先完成关键能力的独立服务化样板与事件驱动样板**。

---

## 2. 总体推进顺序与依赖

### 推荐顺序
1. RAG 服务边界拆分
2. LLM 服务边界拆分
3. 单体兼容层与服务调用切换
4. Kafka/CDC 数据同步样板

### 关键依赖关系
- 依赖第一梯队的正式前端/BFF、真实网关、真实检索和真实集成样板。
- 依赖现有租户隔离、RBAC、审计、配置中心、指标体系、发布门禁。
- Kafka/CDC 样板依赖统一数据域字典与一个真实集成对象。

### 建议节奏
- **里程碑 M1（2~3 周）**：RAG 服务边界与契约固化
- **里程碑 M2（2~3 周）**：LLM 服务边界与成本/治理能力迁移
- **里程碑 M3（2 周）**：单体兼容层切换与灰度验证
- **里程碑 M4（2~3 周）**：Kafka/CDC 样板闭环

---

## 3. T5 拆分 RAG / LLM 为独立服务

- **优先级**：P1
- **现状**：当前 RAG 与 LLM 以内聚 service 形式运行于单体进程中，已有较强治理基线，但故障隔离、独立扩缩容、独立发布能力不足。
- **目标**：拆分 `rag-service` 与 `llm-service`，保留当前单体兼容层与调用契约，形成可灰度切换的服务化样板。
- **负责人角色**：后端架构负责人、AI 平台负责人、平台/DevOps 负责人、测试负责人、SRE
- **预计工期**：15~20 个工作日

### 里程碑
- **M1.1**：明确 RAG / LLM 服务边界与职责清单
- **M1.2**：定义服务契约（HTTP/gRPC 二选一）与错误码体系
- **M1.3**：抽离调用适配层，单体改为通过客户端调用服务
- **M1.4**：增加灰度开关、熔断、超时、回退策略
- **M1.5**：完成独立部署与基础 SLA 验证

### 依赖关系
- 依赖 T7.2 真混合检索、T7.3 Rerank、T7.4 评估看板、T7.6 配置中心、T8.1 指标体系。
- 依赖网关边界已明确，避免服务暴露边界混乱。

### 风险点
- 服务拆分后网络抖动引入额外延迟与失败点。
- 单体和新服务双轨并存期间，契约漂移风险高。
- LLM 成本、租户配额、灰度策略在拆分后可能丢失或重复统计。
- RAG 服务拆分后，Qdrant/ES/DB 连接与租户过滤链路容易出现偏差。

### 实施步骤
1. 明确 `rag-service` 和 `llm-service` 的输入输出协议。
2. 抽取当前 `KnowledgeService`、`LLMGateway`、`RerankService` 的服务客户端层。
3. 引入配置开关：支持 `in-process` / `remote-service` 双模式。
4. 为远程调用补超时、重试、熔断、trace 透传、request_id 透传。
5. 将现有监控、预算、审计、配额逻辑迁移到调用链边界。
6. 通过灰度开关逐步切换真实流量。
7. 在稳定后把单体内部实现降级为 fallback，而不是默认主路径。

### 测试用例
- `rag-service` 在远程模式下完成真实检索请求。
- `llm-service` 在远程模式下返回 usage/token/cost 元数据。
- 单体到远程服务的 trace_id/request_id 透传可见。
- 远程服务超时时，单体可按策略回退。
- 预算/配额/审计/指标在拆分后仍保持一致。
- 灰度切换后至少 1 组真实流量验证成功。

### 验收标准
- `rag-service` 与 `llm-service` 至少一项完成独立部署样板。
- 单体存在兼容层，可切换 `in-process` / `remote-service`。
- 指标、预算、审计、租户过滤链路不丢失。
- 有灰度、回滚与故障回退策略。

### 预计影响范围
- `src/services/`
- `src/infrastructure/`
- `src/api/`
- 部署与网关配置
- 审计、指标、配置中心、发布门禁

## 4. T6 建立 Kafka/CDC 驱动的数据同步样板

- **优先级**：P1
- **现状**：当前系统已有任务、审计、同步方向和主数据口径定义，但没有形成正式的事件总线、CDC 或 Outbox 同步模式。
- **目标**：为 1~2 类关键实体建立 Kafka/CDC 样板链路，形成幂等、重放、重试、DLQ、审计的标准做法。
- **负责人角色**：后端负责人、数据工程负责人、平台负责人、测试负责人、业务分析负责人
- **预计工期**：10~15 个工作日

### 里程碑
- **M2.1**：确定样板实体（建议优先 `Product` 或 `Document`）
- **M2.2**：定义 topic、事件 schema、幂等键、重试策略
- **M2.3**：建立生产者 + 消费者 + DLQ 样板
- **M2.4**：建立状态查询、失败重放、审计记录闭环

### 依赖关系
- 依赖统一数据域字典（T7.5）
- 依赖配置中心（T7.6）
- 依赖发布门禁（T8.2）和指标体系（T8.1）
- 若选 CDC 模式，还依赖数据库表结构与变更策略（Alembic）

### 风险点
- 事件 schema 设计不稳定，后续兼容成本高。
- Outbox/CDC 的 exactly-once 幻觉导致重复消费问题。
- 租户、多系统、重试和补偿的组合复杂度高。
- 外部系统无法提供严格幂等接口时，易形成脏数据。

### 实施步骤
1. 选择样板实体与同步边界。
2. 设计事件 schema、topic 命名、版本字段、幂等键。
3. 实现 Outbox 或 CDC 样板二选一。
4. 建立消费者、重试队列、死信队列和人工重放入口。
5. 为事件链路增加审计、指标和 trace 透传。
6. 输出样板运行手册与异常处理流程。

### 测试用例
- 同一事件重复投递不会造成重复写入。
- 消费失败后进入重试/DLQ，并可人工重放。
- 租户隔离在事件链路中可验证。
- 事件版本升级后旧消费者有兼容策略。
- 至少 1 条端到端样板同步链路成功运行。

### 验收标准
- 至少 1 条 Kafka/CDC 样板链路真实可运行。
- 幂等、重试、DLQ、重放、审计链路完整。
- 事件 schema、topic、版本策略有清晰定义。

### 预计影响范围
- 数据同步链路
- 集成样板
- 审计/指标/任务系统
- 配置中心与部署资源

---

## 5. 第二梯队总体建议

第二梯队的重点不是“追求微服务数量”，而是：

> 先拆关键能力、再做事件驱动样板、保留单体兼容层、确保可灰度回退。

推荐原则：
- 先边界、后拆分
- 先样板、后推广
- 先保兼容、后完全迁移
- 先治理链路完整，再追求规模化拆分

如果资源有限，建议优先次序为：
1. `rag-service` 样板
2. `llm-service` 样板
3. 兼容层与灰度切换
4. Kafka/CDC 样板

这样能最大化降低拆分风险，并为第三梯队的数据平台与 Agent 平台深化打基础。
