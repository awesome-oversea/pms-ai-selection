# 核心问题清单

> 适用范围：当前仓库源码、现有方案文档、子任务文档、验收标准文档
> 用途：作为后续整改、任务拆分、阶段验收的统一问题基线
> 说明：本清单关注“当前真实实现”和“既有方案/验收口径”之间的差距，不等同于目标蓝图

---

## 1. 问题分级说明

- **P0**：阻断主链路、影响部署可用性、导致验收结论失真，必须优先处理
- **P1**：影响系统稳定性、可维护性或后续扩展，需在主链路稳定后尽快处理
- **P2**：优化项或生产化增强项，可在前两级问题稳定后推进

---

## 3. 问题明细

### K01｜方案、验收、源码严重漂移

**级别**：P0  
**现象**：
- 方案和 README 宣称包含企业级多组件架构、多租户、GraphRAG、ERP 闭环、K8s 高可用等能力
- 当前源码主体仍是单体 FastAPI + Jinja2 + 局部 Agent 骨架 + 大量 mock/stub
- 现有验收文档对完成度和测试通过数的描述互相不一致

**影响**：
- 项目状态被高估
- 团队排期容易围绕“假前提”推进
- 用户、研发、测试、管理层对当前阶段认知不一致

**整改建议**：
- 将“当前实现说明”和“目标架构蓝图”拆分维护
- 所有验收标准以当前阶段目标为准，不再直接套用终局方案
- 建立问题基线、任务基线、验收基线三份统一文档

**问题关闭标准**：
- 有单独的“当前系统能力边界”文档
- 当前阶段的任务清单与验收标准仅基于真实可交付范围
- README、方案、验收口径不再互相冲突

---

### K02｜启动、探针、部署链路不可靠

**级别**：P0  
**现象**：
- 应用初始化时对数据库、Redis、Qdrant 等失败场景处理不一致
- Dockerfile 存在不可靠写法
- Compose 中存在“伪高可用”配置
- K8s readiness 路径与应用真实路由不一致
- readiness 语义不真实，依赖失败时仍可能返回 200

**影响**：
- 本地、Docker、K8s 启动行为不一致
- 服务“看似启动成功”，但业务不可用
- 自动化部署和弹性调度无法正确判断实例健康

**整改建议**：
- 统一启动阶段的依赖初始化和失败策略
- 修复 `/health`、`/ready`、`/live` 的语义和路由
- 收敛 Compose 到真实可运行开发环境
- 修正 Dockerfile 和 K8s 探针配置

**问题关闭标准**：
- 本地、Docker、K8s 三种运行方式的启动逻辑一致
- `/ready` 真实反映依赖状态，失败时返回非 2xx
- `docker compose up` 可完成最小可用链路启动

---

### K03｜选品任务主链路以内存态为中心

**级别**：P0  
**现象**：
- 任务、审批、运行态依赖模块级内存字典
- `BackgroundTasks` 承载长任务执行
- 任务取消和恢复能力不完整
- `task_id` 与工作流内部 `session_id` 不一致

**影响**：
- 服务重启后任务状态丢失
- 多实例部署时状态不一致
- 取消、重试、恢复、追踪都不可靠

**整改建议**：
- 建立 Repository + Service 层承接任务状态读写
- 将任务状态、阶段、结果、审批信息持久化
- 统一 task_id 与 workflow session_id
- 第一阶段先完成单体持久化闭环，第二阶段迁移 Worker/Queue

**问题关闭标准**：
- 主链路不再依赖 `_task_store` 等内存字典
- 任务创建、查询、取消、结果获取均以数据库为准
- 服务重启后历史任务可查询
- 存在可验证的取消/失败/完成状态流转

---

### K04｜Web 层与 API 实现细节强耦合

**级别**：P1  
**现象**：
- Web 路由直接依赖 API endpoint 内部变量
- 导入模块时存在目录创建等副作用
- 页面展示逻辑与运行态存储实现绑定

**影响**：
- 接口重构会直接拖垮页面逻辑
- 单元测试边界不清
- 后续拆服务或升级状态存储时改动面过大

**整改建议**：
- 引入 `ApplicationService` / `QueryService`
- Web 与 API 统一通过服务层拿数据
- 移除模块导入副作用

**问题关闭标准**：
- Web 不再 import API endpoint 内部状态变量
- 页面查询走服务层接口
- 模块导入不触发目录创建等非显式副作用

---

### K05｜RAG 上传链路与检索链路断开

**级别**：P0  
**现象**：
- 文档上传后只做了文本读取和 chunk 切分
- 没有真实 embedding、Qdrant 写入、Document/Chunk 持久化
- 检索时新建 retriever，与上传文档状态无关联

**影响**：
- 用户上传的知识库内容不能参与真实检索
- 知识库状态字段失真
- 所有基于知识库的问答/引用都不可靠

**整改建议**：
- 打通上传、切片、向量化、索引、检索闭环
- 文档状态流转明确化：`pending -> processing -> indexed -> failed`
- 查询链路从持久化和向量库恢复，而不是依赖进程内对象

**问题关闭标准**：
- 上传文档后可在数据库和向量库看到对应记录
- 查询能够命中已上传文档内容
- 删除/重建索引后状态能正确更新

---

### K06｜mock / stub / real 能力边界不清

**级别**：P0  
**现象**：
- LLM 网关默认 mock
- 高可用、GraphRAG、ERP 等模块存在 stub/mock 实现
- 部分测试对 stub 本身进行断言，但被用于能力验收

**影响**：
- 无法判断功能是否真实接通
- 调试时难以定位问题在 mock 还是 real provider
- 对外演示和内部验收容易混淆

**整改建议**：
- 统一 Provider 接口：`interface / mock / real`
- 运行模式通过配置显式声明
- API/日志/结果元数据中带上 provider_mode
- mock 仅用于开发和测试，不得直接作为生产验收依据

**问题关闭标准**：
- 每个外部能力都有清晰的 mock 与 real 实现边界
- 配置可切换模式且日志可追溯
- 验收报告明确区分 mock 验证和 real 验证

---

### K07｜依赖声明与真实运行依赖不一致

**级别**：P0  
**现象**：
- `pyproject.toml`、`requirements.txt`、代码导入依赖项不一致
- Kafka 相关库声明与使用不一致
- 部分客户端生命周期不完整

**影响**：
- 开发环境、CI、容器环境安装结果不一致
- 某些功能在本地可运行，部署后失败
- 故障定位成本高

**整改建议**：
- 选定单一依赖源作为准绳
- `requirements.txt` 由标准方式导出，不再手工漂移
- 修复 Kafka / Qdrant / Redis 等客户端生命周期管理

**问题关闭标准**：
- 依赖清单只有一个真实来源
- 新环境安装后能通过基础启动验证
- 关键基础设施客户端具备完整初始化和释放逻辑

---

### K08｜Schema、ORM、接口契约漂移

**级别**：P1  
**现象**：
- 同一字段在 ORM、Pydantic Schema、接口返回中含义不一致
- endpoint 层存在重复 schema 定义
- 某些统计逻辑和实际落库结构不匹配

**影响**：
- 接口返回不稳定
- 前端/测试/统计逻辑易出现隐式错误
- 文档生成与真实响应不一致

**整改建议**：
- 统一模型职责：ORM 管存储，Schema 管接口
- 统一 `go_no_go` 等关键字段结构
- endpoint 层不再内嵌重复 schema

**问题关闭标准**：
- OpenAPI 与真实响应结构一致
- 关键对象仅保留一套对外 schema
- 统计逻辑基于真实存储结构工作

---

### K09｜认证仅停留在基础登录层

**级别**：P1  
**现象**：
- 已存在登录/注册与 JWT 基础能力
- 但未形成角色鉴权、租户隔离、审计追踪闭环

**影响**：
- 无法支撑审批、任务管理、知识库管理等细粒度权限控制
- 企业级场景下存在越权和审计不足风险

**整改建议**：
- 引入角色权限依赖
- 在任务、知识库、报告等对象上加租户/所有者约束
- 对登录、审批、任务取消、文档上传等关键操作记审计日志

**问题关闭标准**：
- 至少具备用户、管理员两级权限边界
- 未授权用户访问受保护接口时返回 401/403
- 审计日志可追踪关键操作人、时间、对象、结果

---

### K10｜测试体系数量大但可信度不足

**级别**：P0  
**现象**：
- 存在与当前实现漂移的测试文件
- 大量测试在 mock 或降级模式下运行
- 多份文档中的测试通过数互相冲突

**影响**：
- “测试通过”不能证明主链路真实可用
- CI 缺少可信门禁
- 质量结论无法作为上线依据

**整改建议**：
- 重建最小可信测试集
- 将测试分层：单元、契约、集成、最小 E2E
- 只保留能代表真实交付质量的核心测试作为门禁

**问题关闭标准**：
- 存在最小可信测试集并纳入 CI
- 核心主链路至少有真实依赖集成测试覆盖
- 测试报告口径统一，能够解释通过/跳过/降级原因

## 4. 建议整改顺序

### 第一优先级：先收敛成“可信单体原型”
1. K01 文档与口径收口
2. K02 启动、探针、部署基线修复
3. K07 依赖统一
4. K03 任务主链路持久化
5. K08 契约收敛

### 第二优先级：补齐“真实功能闭环”
6. K05 RAG 真入库与真检索
7. K06 mock/real provider 分层
8. K04 Web/API 解耦

### 第三优先级：建立“质量与权限边界”
9. K10 最小可信测试集与 CI 门禁
10. K09 RBAC、租户、审计闭环

---

## 5. 本清单的使用方式

- 任务拆分时，所有任务必须引用对应问题 ID
- 每个任务都必须包含明确验收标准
- 阶段验收应以“问题是否关闭”为最终判断依据，而不是仅看代码提交量
- 如未来新增问题，应优先更新本清单，再更新任务清单和验收标准
