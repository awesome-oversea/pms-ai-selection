## 1. 当前系统定位

当前仓库的真实形态不是“只有骨架的 PoC”，也不是“已完成生产化企业中台”。

更准确的定位是：

- 企业级单体 `FastAPI` 主应用
- 具备 `Repository + Service + Worker + BFF + Next.js` 分层骨架
- 选品主链路、利润/ROI、趋势、竞品、报告、飞轮等核心业务闭环已形成第一版
- 多租户、RBAC、审计、Trace、指标、治理状态接口已具备基线能力
- GraphRAG / Triton / Data Lake / Gateway / HA / ERP 集成等能力已形成样板或最小闭环
- 但真实生产环境接入、独立平台化、高可用与最终企业交付仍未完成

## 3. 当前态判定原则

### 3.1 状态口径

- **已实现**：仓库内已有完整代码路径，且具备接口或测试证据
- **部分实现**：已有正式骨架、样板闭环或最小功能，但未达到设计终态
- **未完成**：设计要求明确，但当前仓库中没有足够事实支撑

### 3.2 验收原则

判断某项能力是否可认定完成，至少同时检查：

1. 是否有源码实现
2. 是否有接口/页面/服务链路
3. 是否有测试或构建验证证据
4. 是否依赖真实外部环境

如果仅存在设计文档、历史报告或状态样板，不应直接视为“已完整完成”。

## 6. 使用说明

如果需要判断“某项能力是否完成”，请按以下顺序核查：

1. 查当前态文档
2. 查代码实现
3. 查接口或页面入口
4. 查测试/构建/验证记录
5. 再判断是否达到设计目标

只满足设计文档或历史报告，不应视为当前已完成。
