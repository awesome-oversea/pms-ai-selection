# 设计与代码一致性复核方案

## 1. 目标

针对以下设计、架构、交付、上线、演进、问题分析类文档，与当前代码实现、已完成任务、测试证据之间的一致性进行系统性复核，识别：

- 不一致项
- 能力缺口
- 风险点
- 优先级
- 建议修订清单
- 下一步执行计划

本次复核不是单纯的设计摘要，也不是只做代码扫描，而是按照“文档主张 → 代码事实 → 测试证据 → 结论建议”的方式做交叉核查。

## 3. 检查方法

### 3.1 文档主张拆解

将文档内容拆分为五类主张：

1. 已交付声明
2. 目标态蓝图
3. 分阶段规划
4. 上线/运维要求
5. 企业级治理要求

### 3.2 代码事实映射

每个主张尽量映射到具体实现：

- 文件
- 模块
- 服务
- 接口
- 脚本
- 配置
- 测试

### 3.3 差距分类

问题不会只写“未实现”，而会分成以下类型：

- **文档过时**：代码已完成，但文档未同步
- **代码未达设计**：文档要求明确，但代码缺失或仅有样板
- **实现偏离设计**：代码已做，但方式与设计不同
- **缺测试证据**：代码存在，但缺稳定验证
- **上线/扩展风险**：当前能跑，但企业级风险较高

### 3.4 精确定位原则

尽量精确到：

- 文件
- 模块
- 接口
- 测试
- 脚本
- 文档章节

例如：

- `src/api/v1/endpoints/system.py:/api/v1/slo-status`
- `src/services/data_sync_service.py`
- `scripts/release_quality_gates.py`
- `tests/test_api_integration.py`

### 3.5 优先级评估

按企业级影响进行分级：

- **P0**：影响上线、数据安全、租户隔离、回滚、核心链路
- **P1**：影响企业级闭环完整性
- **P2**：中长期能力缺口
- **P3**：文档整理、命名统一、补说明

## 4. 预计输出结构

最终复核报告建议包含以下结构：

### 4.1 结论摘要
- 当前整体一致性结论
- 当前最严重不一致项
- 当前已完成但文档未同步项
- 当前最大风险点

### 4.2 检查范围与依据
- 纳入文档
- 纳入代码范围
- 使用的测试与验收证据
- 不纳入范围说明

### 4.3 按能力域逐项对照
预计覆盖以下能力域：

1. 系统定位与边界
2. 架构分层
3. 前端/BFF
4. API/Gateway/OpenAPI
5. 选品主流程与 Agent 编排
6. 知识库 / RAG / 检索
7. LLM 路由与治理
8. ERP / 外部系统集成
9. Kafka / CDC / 事件链路
10. 数据湖 / 批流样板
11. 压测 / SLO / 门禁
12. 迁移 / 上线 / 回滚
13. 运维 / Runbook / Oncall / SLA

每个能力域都输出：
- 设计主张
- 当前实现
- 证据位置
- 一致性结论
- 缺口/风险
- 建议修订动作

### 4.4 不一致项清单
建议采用表格形式：

- 编号
- 文档来源
- 设计主张
- 当前代码事实
- 证据位置
- 偏差类型
- 风险等级
- 优先级
- 修订建议

### 4.5 缺口与风险清单
强调企业级仍未完成或仍处于样板态的能力：

- 缺口
- 当前状态
- 影响范围
- 风险
- 优先级
- 推荐路径

### 4.6 文档修订清单
明确哪些文档需要调整口径：

- 哪些应标为目标态
- 哪些应标为当前已交付态
- 哪些章节存在冲突或重复
- 哪些 deliverables / phase4 文档与当前实现漂移

### 4.7 下一步执行计划
分为两部分：

- 先修文档口径
- 再补代码缺口
- 最后统一复验

## 5. 预计重点关注项

本次复核预计重点关注以下高风险区域：

### 5.1 目标态与已交付态混淆
尤其需要核实：

- 设计方案
- architecture 文档
- deliverables 文档
- phase4 上线文档

是否把蓝图能力误写为当前已交付能力。

### 5.2 已完成能力未同步回文档
重点核实以下已完成但可能未同步的能力：

- `slo-status`
- `data-lake/status`
- Agent 平台接口
- Kafka/CDC 样板闭环
- OMS 集成样板
- 网关治理状态与配置样板

### 5.3 上线方案与当前门禁口径漂移
重点检查：

- `release_quality_gates.py`
- `perf-smoke`
- 回滚与迁移门禁
- SLO/性能口径

### 5.4 架构蓝图与当前单体治理基线差异
当前代码更接近“企业级治理型单体 + 样板拆分能力”，而不是完全的目标态分布式平台。需要明确：

- 现状
- 演进态
- 目标态

### 5.5 样板能力是否被误写为正式全面生产化
重点核实：

- 网关治理
- OMS 集成
- Data Lake
- Agent 平台
- Kafka/CDC

是否应明确标注为“样板已完成”，而不是“全面生产化已完成”。

## 6. 执行顺序

建议按以下顺序执行正式复核：

### 第 1 步：文档盘点
建立“文档主张索引表”。

### 第 2 步：代码事实盘点
建立“当前实现事实索引表”。

### 第 3 步：测试证据盘点
建立“测试/接口/脚本证据索引表”。

### 第 4 步：差距与风险归并
形成：
- 不一致项
- 缺口
- 风险
- 优先级
- 修订建议

### 第 5 步：输出正式结果文档
建议至少输出：

1. `docs/设计与实现一致性复核报告.md`
2. `docs/设计文档不一致项与修订清单.md`

如需要，可再补：

3. `docs/一致性修订执行计划.md`
