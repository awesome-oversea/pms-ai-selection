# 架构演进路线图

> 适用范围：从当前项目实现出发，对系统从“当前单体原型”演进到“企业级分层服务体系”的路径进行分阶段说明
> 关联文档：
> - `选品系统设计方案.md`
> - `docs/architecture/`
> - `docs/deliverables/`
> - `docs/企业级问题清单.md`
> - `tasks/Phase 5~8 企业级优化任务清单.md`
> 说明：本路线图不是终局万能蓝图，而是“阶段性边界说明 + 架构演进顺序说明”，用于减少过度设计和提前拆分

---

## 1. 路线图总览

系统建议按以下四个阶段演进：

1. **当前单体原型**
2. **企业单体**
3. **平台化**
4. **分层服务化**

每个阶段都必须回答四个问题：
- 当前阶段保留什么
- 当前阶段新增什么
- 当前阶段明确不做什么
- 什么信号说明应该进入下一阶段

---

## 2. 阶段一：当前单体原型

### 2.1 阶段定义

这是当前项目已经达到的状态：
- 单体 FastAPI 应用为主
- 已完成任务主链路、知识库主链路、基础 RBAC、基础审计、最小可信测试、CI 基线
- PostgreSQL / Qdrant / LLMGateway 已有真实可验证链路

### 2.2 当前阶段的核心特征

- **系统形态**：单体应用
- **执行模式**：API 进程 + 进程内任务执行（已预留 Worker 抽象）
- **数据存储**：PostgreSQL + Qdrant + Redis（有限使用）
- **RAG 能力**：具备基础上传、切片、向量化、检索能力
- **权限能力**：基础认证与管理员保护
- **质量基线**：最小可信测试、CI、健康探针、阶段验收清单

### 2.3 当前阶段保留什么

- 单体部署与开发效率
- 一套代码仓库内收敛主链路
- 基于数据库和向量库的真实可验证路径
- 面向功能正确性的最小测试体系

### 2.4 当前阶段明确不做什么

- 不急于拆微服务
- 不急于引入大量中间件（Flink / Spark / Ray / Airflow）
- 不急于做复杂网关与前后端大分裂
- 不把目标态蓝图直接当作当前实现

### 2.5 当前阶段的主要边界

- 缺少多租户
- 缺少正式 RBAC / ABAC
- 缺少持久化审计中心
- 缺少 Worker 平台
- 缺少企业发布治理

### 2.6 进入下一阶段的信号

当满足以下任一条件时，应进入“企业单体”阶段：
- 需要支持多个客户、事业部、品牌或业务线
- 需要正式权限体系和合规审计
- 需要更强的任务执行隔离和资源治理
- 需要企业级发布与运维要求

---

## 3. 阶段二：企业单体

### 3.1 阶段定义

企业单体不是“更大的单体”，而是：
> 在保持单体部署/研发效率的前提下，把企业级治理能力补齐。

这是最推荐的下一阶段。

### 3.2 该阶段的目标

- 引入多租户
- 建立正式 RBAC / ABAC
- 建立持久化审计中心
- 建立 request_id / trace_id 链路
- 将任务执行从 API 逻辑中解耦为平台能力，但仍可部署在同一产品体系内
- 建立统一 API 契约

### 3.3 该阶段保留什么

- 单仓库开发体验
- 核心服务仍以单体应用为主
- 共享数据库（但严格租户边界）
- 一致的产品语义与部署运维入口

### 3.4 该阶段新增什么

- `tenant_id` 体系
- 正式权限模型
- 持久化审计
- request_id / trace_id
- Worker 执行模块
- 使用 Alembic 的迁移治理
- 统一 API 契约与错误码体系

### 3.5 该阶段明确不做什么

- 不立即拆成一堆微服务
- 不急于把每个域都独立数据库
- 不急于做过重的数据平台
- 不为了“服务化”而牺牲研发效率

### 3.6 企业单体的边界说明

#### 边界一：部署仍可相对统一
- API / Worker 可以逻辑解耦，但不一定需要完全独立团队维护

#### 边界二：数据仍可共享，但必须受治理
- 共享 PostgreSQL 不代表共享查询权限
- 所有核心查询必须有租户和权限边界

#### 边界三：领域要开始清晰，但未必要立刻分服务
- Selection / Knowledge / Report / Audit / Auth 先做清晰模块边界

### 3.7 进入下一阶段的信号

满足以下信号时进入“平台化”：
- 任务执行、模型调用、知识平台开始成为多个业务域共享能力
- 不同团队开始同时开发不同域能力
- 成本治理、队列治理、配置治理变得复杂
- API、任务、知识库已经不再只是单产品功能，而开始成为共用平台能力

---

## 4. 阶段三：平台化

### 4.1 阶段定义

平台化不是“拆服务的同义词”，而是：
> 把原来作为业务功能存在的能力，升级为可治理、可共享、可复用的平台能力。

### 4.2 平台化重点域

建议优先平台化的能力：
- **任务平台**：Worker、重试、死信、优先级、资源治理
- **模型平台**：Prompt、路由、预算、配额、降级、灰度
- **知识平台**：版本化、索引重建、混合检索、rerank、评测
- **审计平台**：统一事件记录、回查、导出
- **配置平台**：配置中心、特性开关、策略中心

### 4.3 该阶段保留什么

- 业务系统仍可共享一部分单体 API 层
- 数据仍未必完全分库分服务
- 一些平台能力可以先以模块化/独立进程形态存在，而不是立即独立业务服务

### 4.4 该阶段新增什么

- Worker / Queue / Retry / Dead-letter 平台
- 模型调用治理平台
- 真混合检索与 rerank 平台
- 配置中心 / Feature Flag
- 统一成本、指标、配额和审计能力

### 4.5 该阶段明确不做什么

- 不在平台化初期就把所有能力都做成单独“组织级大中台”
- 不在没有稳定接口前过早做跨团队强依赖
- 不把平台化理解成“技术团队可以脱离业务”

### 4.6 平台化的边界说明

#### 边界一：共享能力要先标准化再抽平台
- 没有稳定 API 契约、事件模型、权限模型前，不要抽平台

#### 边界二：平台首先服务于现有业务闭环
- 平台不是为了好看，而是为了减少业务重复建设和治理成本

#### 边界三：平台化优先抽“高复用 + 高治理价值”的能力
- 任务平台
- 模型平台
- 知识平台
- 审计平台
- 配置平台

### 4.7 进入下一阶段的信号

满足以下条件时，进入“分层服务化”：
- 多个团队独立维护不同平台和业务域
- 发布节奏不一致，互相影响明显
- 某些平台域（如任务、模型、知识）需要独立扩容与独立 SLA
- 数据域边界清晰，服务间契约稳定

---

## 5. 阶段四：分层服务化

### 5.1 阶段定义

分层服务化不是“为了微服务而微服务”，而是：
> 在平台化稳定后，按业务域和平台域进行有边界的服务拆分，使团队、部署、扩缩容、SLA 可以独立演进。

### 5.2 推荐拆分层次

#### 第一层：入口层
- API Gateway
- BFF
- 对外开放 API

#### 第二层：业务层
- 选品业务服务
- 报告服务
- 审批/工作台服务
- ERP 集成服务

#### 第三层：平台层
- 任务平台服务
- 模型网关服务
- 知识检索服务
- 审计服务
- 配置中心服务

#### 第四层：数据与基础设施层
- PostgreSQL
- Qdrant
- Redis
- Kafka
- ES/OpenSearch
- 指标与日志系统

### 5.3 该阶段保留什么

- 对业务方稳定的统一产品入口
- 清晰的数据域边界
- 统一的租户、权限、审计、追踪协议

### 5.4 该阶段新增什么

- 服务级 SLA
- 服务级部署和扩缩容
- 服务级团队 ownership
- 服务级观测和容量治理
- 服务级发布与回滚策略

### 5.5 该阶段明确不做什么

- 不把低价值、小体量、低复用模块强行拆服务
- 不在没有 owner、没有观测、没有契约治理的情况下拆服务
- 不让服务化先于租户、权限、审计、迁移、观测体系

### 5.6 分层服务化的边界说明

#### 边界一：拆分必须以稳定契约为前提
- API 契约
- 事件契约
- 数据口径
- 权限协议
- 审计协议

#### 边界二：拆分必须以团队治理能力为前提
- 没有 owner、SLA、值班责任，不应贸然拆分

#### 边界三：拆分必须以发布收益为前提
- 若拆分不能降低发布耦合和运行风险，就不值得拆

---

## 6. 各阶段的关键差异对照

| 维度 | 当前单体原型 | 企业单体 | 平台化 | 分层服务化 |
|---|---|---|---|---|
| 目标 | 证明主链路可用 | 建立企业边界 | 抽共享治理能力 | 支撑团队与服务独立演进 |
| 多租户 | 基本无 | 正式引入 | 平台能力普遍租户化 | 服务间统一租户协议 |
| 权限 | 基础管理员 | 正式 RBAC/ABAC | 平台统一授权模型 | 服务间授权协同 |
| 审计 | 基础/局部 | 持久化中心 | 审计平台 | 服务化审计集成 |
| 任务执行 | 进程内 | Worker 初步平台化 | 任务平台共享 | 任务平台独立服务 |
| 模型调用 | 功能性网关 | 有治理能力 | 模型平台 | 独立模型治理服务 |
| 知识库 | 功能闭环 | 企业可用 | 知识平台 | 检索平台服务化 |
| 发布 | 单体发布 | 企业单体治理 | 多模块协同发布 | 服务级独立发布 |
| 运维 | 研发主导 | 有 Runbook | 平台治理 + 看板 | SRE 化与 SLA 化 |

---

## 7. 推荐演进顺序

### 推荐顺序
1. 当前单体原型 → 企业单体
2. 企业单体 → 平台化
3. 平台化 → 分层服务化

### 不推荐顺序
- 当前单体原型 → 直接微服务化
- 当前单体原型 → 直接上大数据/大中台全家桶
- 未补租户/权限/审计/追踪 → 先拆服务

原因：
- 过早拆分会把现有未治理问题放大
- 企业级问题的核心不是“服务数量”，而是“边界和治理能力”

---

## 8. 每阶段完成判断

### 当前单体原型完成判断
- 主链路真实可验证
- 最小可信测试与 CI 存在
- 不再依赖大面积内存态

### 企业单体完成判断
- 多租户、正式权限、审计、request_id/trace_id、迁移治理已建立
- 任务执行已和 API 解耦
- 统一 API 契约已落地

### 平台化完成判断
- 任务、模型、知识、审计、配置至少有三项真正平台化
- 存在共享能力、共享治理、共享指标
- 不同业务域可复用平台能力

### 分层服务化完成判断
- 服务边界稳定
- 契约稳定
- 团队 ownership 稳定
- 发布、SLA、值班责任清晰
- 服务拆分显著带来独立扩缩容和发布收益

---

## 9. 最终建议

### 9.1 你当前最应该做的不是“继续扩功能”
而是：
- 先进入 **企业单体** 阶段
- 把租户、权限、审计、追踪、迁移、任务治理补齐

### 9.2 只有企业边界稳了，平台化才有意义
否则平台化会变成“把混乱抽成公共混乱”。

### 9.3 只有平台稳定后，分层服务化才值得做
否则会把原本能在一个仓库里快速修的问题，升级成跨团队、跨服务、跨发布窗口的复杂问题。

---

## 10. 反模式提醒

以下做法不建议采用：

- 在未完成多租户、权限、审计、追踪前直接拆微服务
- 在没有稳定 API 契约前先做大规模平台抽象
- 在没有真实发布治理前引入大量服务和中间件
- 在没有 Owner / SLA / Runbook 的情况下做服务独立化
- 用目标态蓝图代替当前阶段交付事实

这些反模式会导致：
- 研发复杂度上升但交付价值不提升
- 运维与排障难度陡增
- 架构分层看似高级，实际治理能力更弱

---

## 11. 结论

推荐的企业级演进路径是：

```text
当前单体原型
  → 企业单体
  → 平台化
  → 分层服务化
```

其中：
- **下一步立即要做的是企业单体，而不是直接微服务化**
- **企业单体阶段的完成质量，决定后续平台化和服务化的成本与成败**
