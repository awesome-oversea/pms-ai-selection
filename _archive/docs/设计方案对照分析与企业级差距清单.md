# 选品系统设计方案对照分析与企业级差距清单

## 1. 结论摘要

《选品系统设计方案.md》明确声明其口径是**目标态 / 规划态企业级蓝图**，而当前仓库代码已经完成的是一套**可运行、可测试、具备企业级治理基线的单体平台**。  
结论上，当前代码**并未完整实现设计方案中的最终形态**，但已经在以下方面完成了较高质量的落地：

- 多租户、RBAC、审计、trace 基线
- Worker 化任务平台、重试/死信/并发治理
- 模型预算、Prompt/Policy 治理、统一 API 契约
- 知识库版本化、混合检索、评测、配置中心
- 发布门禁、接口治理、Alembic 治理、Runbook / SLA / Oncall

从企业级项目角度看，当前系统更接近：

> **“从原型收敛到企业级单体平台基线”**，而不是设计方案中的 **“多系统集成、分层网关、AI中台、数据湖、流批一体、分布式智能决策平台”**。

### 2.2 与设计方案相比的明显缺口

#### A. 前端/UI 与 BFF 目标态未落地
设计方案要求：
- Next.js 14 / Vue3 / Element Plus / SSE / WebSocket 的多层前端体系

当前现状：
- 仅有 `src/web/routes.py` + `web/templates` 的服务端模板页面
- 不存在真正意义上的 Next.js BFF / SSR 前端工程

影响：
- 无法支撑设计方案中的复杂工作台、实时看板、运营后台分层

#### B. 网关层未实现真实 Kong / API Gateway 治理
设计方案要求：
- Kong Gateway 集群、统一鉴权、灰度、限流、开放接口治理

当前现状：
- 代码内有接口治理元数据和限流中间件
- 但没有真实 Kong / APISIX / 云网关落地

影响：
- 当前接口治理仍主要停留在应用内规则，而非基础设施级网关治理

#### C. AI中台未形成独立服务层
设计方案要求：
- `llm-service / rag-service / agent-service / embedding-service` 独立中台能力

当前现状：
- 这些能力在代码中存在，但主要以内聚在单体应用内的 service / infrastructure 形式存在
- 没有真正拆分为独立部署、独立 SLA、独立伸缩服务

影响：
- 扩容边界、故障隔离、独立发布能力弱于目标态

#### D. 数据中台 / 流批一体未落地
设计方案要求：
- Kafka + Flink + Spark + Iceberg/Hudi + Elasticsearch + Milvus

当前现状：
- Kafka、Triton、Ollama、Istio 等多为配置或预留实现
- Elasticsearch / OpenSearch、Milvus、Flink、Spark、Iceberg/Hudi 并未形成真实生产链路
- 当前知识检索主要是 Qdrant + BM25 本地融合

影响：
- 离目标态“统一数据湖 + 实时流处理 + 十亿级检索”的差距仍然很大

#### E. ERP / 内部系统集成未形成闭环
设计方案要求：
- OMS / WMS / SCM / CRM / FMS / BI / PaaS 等系统深度集成

当前现状：
- 数据模型与方向有字典化定义
- 但未看到真实 200+ 企业接口平台、CDC、Kafka 事件流闭环

影响：
- 当前系统仍是“选品平台”，尚未成为“企业利润中枢”

#### F. 多 Agent 复杂编排仍是最小版本
设计方案要求：
- AutoGen4j / LangGraph4j / CrewAI / Dify / Langflow 等多框架协同

当前现状：
- 有 `SelectionMaster`、Agent 基础类和部分多阶段处理
- 但大多仍是 Python 内部编排，不是设计蓝图中的多框架融合编排体系

影响：
- 复杂推理、可视化编排、任务断点恢复、跨语言 Agent 平台能力尚不足

## 3. 企业级视角下当前主要问题清单

### P1. 架构目标与当前实现仍存在层级差
- 设计方案是**分层平台 + AI中台 + 数据中台 + 网关集群 + 多系统集成**
- 当前实现是**企业级单体收敛架构**
- 风险：容易在沟通中把“蓝图”误判为“已交付能力”

### P2. 基础设施级能力多为预留而非真实生产闭环
包括：
- Kong
- ES/OpenSearch
- Milvus
- Flink/Spark/Iceberg/Hudi
- Triton / Istio / Airflow

风险：
- 真实容量、真实 HA、真实成本模型无法按蓝图口径承诺

### P3. 前端/BFF 与运营工作台能力明显不足
- 缺少真正的 Next.js / BFF 层
- 缺少前后端职责边界清晰的产品化界面系统

风险：
- 企业级运营、客服、审核、财务、管理员等角色难以形成正式工作台

### P4. 外部系统集成与主数据同步尚停留在模型层/规划层
- 数据域字典已建立
- 但未落真实同步引擎、冲突处理、CDC、主数据主权治理

风险：
- 一旦接入真实 ERP / BI，数据口径与同步一致性会成为新瓶颈

### P5. 非功能能力虽有基线，但尚未达到“大规模企业生产态”
当前已有：
- 发布门禁
- 指标体系
- Runbook
- Alembic 基线

但缺：
- 多环境发布流水线矩阵
- 混沌演练
- 多 AZ 压测与容量模型
- 真实 SLO 统计闭环

## 4. 建议

### 短期建议（1~2个版本）
1. 把“蓝图能力”和“当前已实现能力”彻底分文档维护
2. 先补真正的 BFF / 前端工程，而不是继续只做后端治理
3. 落一个真实 ES/OpenSearch 检索链路，缩小与设计方案差距
4. 把至少一个 ERP 系统做成真实双向集成样板

### 中期建议（3~6个版本）
1. 拆出独立 `rag-service / llm-service`
2. 补真实 API 网关（Kong 或等价方案）
3. 引入 CDC / Kafka 驱动的统一数据同步链路
4. 补多环境发布矩阵与容量压测基线

### 长期建议（6个版本以上）
1. 建 AI 中台与数据中台正式边界
2. 建统一主数据治理与数据湖体系
3. 建真正的企业经营驾驶舱与利润闭环
4. 形成“选品 → 采购 → 库存 → 销售 → 财务”的闭环平台

## 5. 未来进阶方向

### 方向 1：从企业级单体走向平台化服务拆分
目标：
- 单体治理基线 → 多服务平台

### 方向 2：从知识平台走向数据智能平台
目标：
- Qdrant + BM25 → ES + 向量 + 数据湖 + 评测平台

### 方向 3：从任务编排走向 Agent 平台
目标：
- 任务执行器 → 多 Agent 编排平台 → 可视化调度与运营

### 方向 4：从选品工具走向经营决策系统
目标：
- 只做选品分析 → 驱动利润闭环的经营中枢

## 7. 最终判断

从企业级项目要求角度：

- **当前代码不是《选品系统设计方案》最终形态的完整实现**
- **但已经完成了一个质量较高的企业级收敛基线**
- 如果把设计方案视为“目标态蓝图”，当前仓库可以判断为：

> **已完成目标态的治理底座与平台基线，但尚未完成目标态的分布式架构、中台拆分、数据中台和多系统集成闭环。**

这意味着当前最合理的策略不是推翻重来，而是：

> **以当前已完成的企业级基线为底座，分阶段向设计方案目标态演进。**
