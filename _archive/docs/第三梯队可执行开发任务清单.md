# 第三梯队可执行开发任务清单

## 1. 说明

本文基于 `docs/企业级进阶任务清单.md` 的“第三梯队”任务进一步拆解，目标是把平台深化阶段的工作转换为可执行、可排期、可验收的开发清单。  
本梯队聚焦以下 3 项：

1. 构建数据湖/批流一体样板  
2. 建立容量压测与 SLO 统计闭环  
3. 提升 Agent 平台能力

第三梯队不再只是“功能补齐”，而是面向：**经营分析、容量治理、智能平台化** 的企业长期能力建设。

---

## 2. 总体推进顺序与依赖

### 推荐顺序
1. 容量压测与 SLO 统计闭环
2. 数据湖/批流一体样板
3. Agent 平台能力提升

### 关键依赖关系
- 依赖第二梯队服务拆分与事件驱动样板，才能把数据和服务边界稳定下来。
- 容量与 SLO 统计是第三梯队其它工作的安全前提。
- 数据湖样板可为经营分析、评测、模型优化和 Agent 运行分析提供统一数据基础。
- Agent 平台深化依赖可观测性、数据资产与服务边界已具备。

### 建议节奏
- **里程碑 M1（2 周）**：容量压测与 SLO 基线
- **里程碑 M2（3~4 周）**：数据湖/批流样板
- **里程碑 M3（3~4 周）**：Agent 平台能力深化

---

## 3. T7 构建数据湖/批流一体样板

- **优先级**：P2
- **现状**：当前已有主数据字典、评估看板、检索链路和业务实体，但缺统一数据资产层与批流一体处理样板。
- **目标**：建立最小数据资产平台样板，为经营分析、AI 评测与跨系统数据治理提供统一基础。
- **负责人角色**：数据平台负责人、后端负责人、平台负责人、BI/分析负责人、测试负责人
- **预计工期**：15~20 个工作日

### 里程碑
- **M1.1**：定义样板数据域与数据资产清单
- **M1.2**：完成离线快照样板链路
- **M1.3**：完成小规模实时流样板链路
- **M1.4**：输出字段口径、分区策略与消费说明

### 依赖关系
- 依赖第二梯队事件驱动同步样板
- 依赖主数据模型与字段字典（T7.5）
- 依赖监控指标和发布门禁基础

### 风险点
- 数据资产平台容易过度设计，投入过大。
- 数据口径不统一会导致后续 BI 与评估失真。
- 流批双链路同时建设会增加维护复杂度。

### 实施步骤
1. 选择 1~2 个样板数据域（如检索评测数据、任务执行数据）。
2. 定义样板表结构、分区字段、主键与时间字段标准。
3. 搭建离线快照链路（例如对象存储/Parquet/Iceberg 样板）。
4. 搭建实时增量链路（基于 Kafka 或等价事件输入）。
5. 输出字段字典、口径说明、消费样例和异常处理策略。

### 测试用例
- 离线快照数据可成功落入目标存储。
- 实时样板链路可消费至少一类事件。
- 同一实体在流批链路中的主键和时间语义一致。
- 至少 1 条下游消费查询能获取正确结果。

### 验收标准
- 至少 1 类业务数据形成可追溯数据资产样板。
- 至少 1 条实时样板链路可运行。
- 存在字段口径、分区与消费说明。

### 预计影响范围
- 数据平台与存储层
- 事件链路
- BI/分析消费
- 评估与经营分析能力

## 4. T8 建立容量压测与 SLO 统计闭环

- **优先级**：P2
- **现状**：当前已有发布门禁、指标体系、看板和基础告警，但缺正式容量基线、压测结论和 SLO 周期统计。
- **目标**：形成从压测脚本、容量基线、SLO 计算到告警策略的完整闭环。
- **负责人角色**：性能负责人、后端负责人、平台/SRE、测试负责人
- **预计工期**：10~15 个工作日

### 里程碑
- **M2.1**：确定核心压测场景
- **M2.2**：建立压测脚本与报告模板
- **M2.3**：形成容量结论和 SLO 指标口径
- **M2.4**：将 SLO 统计接入看板与告警

### 依赖关系
- 依赖当前三层指标体系（T8.1）
- 依赖发布门禁（T8.2）
- 依赖服务边界已经较稳定

### 风险点
- 压测场景与真实流量差异太大，结论失真。
- 缺真实依赖环境时，压测结果偏乐观。
- SLO 指标设计不合理会导致“可测不可用”现象。

### 实施步骤
1. 确定核心链路：任务创建、知识检索、LLM 路由、文档上传等。
2. 编写压测脚本并沉淀报告模板。
3. 输出单实例容量结论、推荐副本数和扩缩容阈值。
4. 定义 SLI/SLO（可用性、延迟、错误率、恢复时间）。
5. 将 SLO 统计结果接入统一看板和告警。

### 测试用例
- 至少 1 组 API 压测脚本可运行。
- 至少 1 组 Worker/异步链路压测可运行。
- 能输出 p95/p99、错误率、吞吐量等指标。
- 有明确容量结论和告警阈值。

### 验收标准
- 至少 1 套压测场景可重复执行。
- 有容量基线和扩容建议。
- 有 SLO 指标定义、统计方式和告警联动。

### 预计影响范围
- `scripts/`
- 指标体系与看板
- 发布门禁
- 平台/SRE 运维流程

## 5. T9 提升 Agent 平台能力

- **优先级**：P2
- **现状**：当前多 Agent 工作流仍以 Python 内部编排为主，具备治理基础，但距离可视化、断点恢复、策略发布、运行分析平台仍有差距。
- **目标**：把当前 Agent 能力从“任务型编排”提升到“平台型能力”。
- **负责人角色**：AI 平台负责人、后端负责人、前端负责人、平台负责人、测试负责人
- **预计工期**：15~20 个工作日

### 里程碑
- **M3.1**：建立 Agent 运行拓扑与节点元数据模型
- **M3.2**：支持断点恢复、人工介入、失败重试
- **M3.3**：建立 Agent 运行观测与操作台
- **M3.4**：支持策略发布与版本化回滚

### 依赖关系
- 依赖现有任务平台化、审计、trace、指标、配置中心
- 依赖前端/BFF 与接口治理已到位
- 依赖数据资产与质量评估体系支撑运行分析

### 风险点
- Agent 平台功能边界过大，容易和现有任务系统重叠。
- 可视化编排若先行，会导致维护复杂度过高。
- 恢复/重试/人工介入与状态机语义容易冲突。

### 实施步骤
1. 明确 Agent 平台与任务平台的边界。
2. 抽象 Agent 节点、运行图、状态转换与恢复语义。
3. 建立操作台视图：运行中、失败、重试、人工介入。
4. 增加 Agent 策略版本化、灰度发布与回滚能力。
5. 补充运行诊断、失败原因聚类和行为分析。

### 测试用例
- 至少 1 个多 Agent 任务支持失败后恢复。
- 至少 1 个 Agent 策略版本可切换与回滚。
- 至少 1 个运行拓扑可被页面/接口消费。
- 失败任务具备人工介入与审计记录。

### 验收标准
- 至少支持任务恢复、运行诊断和策略切换中的两项真实闭环。
- 存在 Agent 运行操作台或等价接口。
- 运行分析可追踪到节点、阶段、错误与回滚。

### 预计影响范围
- `agents/`
- `selection_service`
- 前端/BFF
- 配置中心、审计、指标、任务状态机

---

## 6. 第三梯队总体建议

第三梯队应遵循：

> 先做容量与可观测闭环，再建设数据资产样板，最后把 Agent 从“功能”升级为“平台”。

推荐原则：
- 先可测、后扩展
- 先数据基础、后智能运营
- 先运行保障、后复杂编排
- 先样板、后平台化推广

若资源有限，建议优先次序为：
1. 容量压测与 SLO 统计闭环
2. 数据湖/批流一体样板
3. Agent 平台能力提升

这样能够避免平台深化过程中失去性能和运维可控性，也能为后续真正的数据中台/AI 中台演进打下稳固基础。
