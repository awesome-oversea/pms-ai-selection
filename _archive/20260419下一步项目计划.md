## 1. 计划目标

本阶段不再继续横向扩展模块数量，而是围绕一个真实业务样板，完成从“可演示”到“可重复交付”的收口。

计划总目标如下：

1. 以“户外储能电源”作为样板场景，打通真实黄金链路。
2. 将关键运行模式从 `mock / compatible` 收口到 `local-real / remote-service`。
3. 修复当前可信回归中的阻塞失败项，恢复高稳定基线。
4. 建立真实依赖、前后端联调、验收证据三位一体的交付方式。

## 2. 当前基线

当前项目已经具备较完整的架构与业务覆盖，但下一阶段的主问题不是“功能不够多”，而是“真实运行证据不够完整”。

当前计划以以下事实作为基线：

- 主链路、Agent 编排、审批、反馈、报告、租户、审计、工作台能力已有落点。
- 核心能力仍混合存在 `mock / compatible / local-real / remote-service` 多种模式。
- 可信回归执行结果为 `240` 项中 `229` 通过、`11` 失败。
- 当前失败主要集中在 OCR 模式契约、样本路径健壮性、real Postgres / real Qdrant 持久化稳定性。

## 3. 本阶段交付定义

本阶段以“真实闭环收口”为唯一主线，最终必须交付以下结果：

### 3.1 业务交付

- 可以创建“户外储能电源”选品任务
- 可以触发 Agent 执行并产出推荐、风险、利润结果
- 可以完成审批、采纳、执行反馈、再评分
- 可以导出正式报告并在工作台查看结果

### 3.2 技术交付

- 关键状态接口能返回当前模式、fallback、依赖健康度
- Postgres、Redis、Qdrant、OpenSearch、Kafka、Kong 单节点栈可重复启动
- `SelectionTaskService` 开始按职责拆分，降低后续改动风险
- 前后端联调链路可以稳定跑通

### 3.3 质量交付

- 当前 `11` 个阻塞失败项完成修复或压缩到非阻塞范围
- 建立结构层、本地运行层、真实依赖层三层验证体系
- 新增主链路 Playwright smoke
- 形成阶段验收证据包与运行手册

## 9. 风险与控制措施

| 风险 | 表现 | 控制措施 |
| --- | --- | --- |
| 模式口径混乱 | 看似完成，实际仍运行在 mock | 模式矩阵、状态接口、环境模板同步收口 |
| 真实依赖不稳定 | Postgres / Qdrant / Kafka 随机失败 | 先做单节点基线和专项回归 |
| 服务持续膨胀 | `SelectionTaskService` 成为后续改动阻塞点 | 按职责拆分，优先拆核心闭环能力 |
| 前端验收失真 | 页面存在但交互不通 | 强制纳入 Playwright 联调 smoke |
| 过早扩展基础设施 | K8s / HA 投入增加但主链路未闭环 | 本阶段坚持单节点基线优先 |

## 10. 计划结论

下一步项目计划的核心不是“继续扩面”，而是“围绕真实黄金链路完成阶段收口”。

只要本阶段完成以下三件事，项目就能从“能力铺设阶段”进入“可交付基线阶段”：

1. 跑通真实链路
2. 修复关键阻塞
3. 固化运行证据

完成这一步后，再进入 K8s / HA、多模态、GraphRAG、生产级扩展，投入才是有效的。
