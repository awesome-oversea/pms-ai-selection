## 1. 验收框架与口径

### 1.1 双口径原则

本验收标准同时区分两个判断层：

1. **企业可交付判断**
   - 判断项目是否已从“样板级平台能力 + 单体治理基线”进入“企业可交付阶段”。
2. **设计方案终态判断**
   - 判断项目是否已经基本对齐 `选品系统设计方案.md` 的企业级终态蓝图。

### 1.2 结果分级

| 结果级别 | 定义 | 是否可计入企业级通过 |
|---|---|---|
| `enterprise-pass` | 真实环境、真实系统、真实数据、真实运维链路下通过，且具备测试、监控、审计、回滚证据 | 是 |
| `real-pass` | 真实链路通过，但规模、运维、稳定性或治理证据尚未齐全 | 仅可计入阶段性通过，不可直接认定企业终态完成 |
| `sample-pass` | 样板链路、模拟链路、降级模式或实验环境下通过 | 否 |
| `skipped` | 因外部环境、第三方系统、资源未到位而未执行 | 否 |
| `failed` | 未满足功能、性能、安全或交付要求 | 否 |

### 1.3 证据要求

每个验收项至少提供以下证据中的 3 类：

1. **代码证据**：实现文件、配置文件、Schema、脚本、测试代码
2. **执行证据**：接口返回、日志、截图、录屏、压测报告、部署记录
3. **治理证据**：审计日志、Trace ID、告警记录、回滚记录、变更记录
4. **业务证据**：真实业务数据、真实系统联调记录、业务方验收记录

### 1.4 不允许再混淆的事项

以下情形不允许再被认定为“企业级通过”：

- 配置文件存在 ≠ 已完成生产部署
- 接口存在 ≠ 已完成企业系统联调
- 样板闭环可演示 ≠ 企业闭环可交付
- 前端骨架存在 ≠ 正式工作台可交付
- `sample-pass` / `mock-pass` ≠ 企业级完成
- 单次手工演示通过 ≠ 持续稳定运行

## 2.2 企业可交付通过条件

达到“企业可交付阶段”，至少需要同时满足：

1. M0 ~ M6 全部通过
2. 全部 P0 验收项达到 `enterprise-pass`
3. 全部 P1 验收项至少达到 `real-pass`，其中关键平台项应达到 `enterprise-pass`
4. 不存在 `failed` 的高优先级缺陷
5. 不存在未经审批的 `skipped` 项
6. 已形成上线、回滚、值守、审计、告警、交付文档闭环

## 2.3 设计方案终态对齐通过条件

若要认定“基本达到 `选品系统设计方案.md` 目标态”，还需要：

1. 全部 P0、P1、P2 核心验收项达到 `enterprise-pass`
2. 企业级业务闭环、平台化架构、关键技术栈均进入真实运行
3. 不再依赖 memory fallback、mock graph、手工样板联调作为主路径
4. 具备多环境、HA、SLO/SLA、安全治理和运维值守能力

## 3. 技术栈验收原则

## 3.1 必须实装的技术能力

下列能力属于企业级验收必核项，不得只以“样板代码存在”认定通过：

| 技术域 | 企业级要求 | 验收要求 |
|---|---|---|
| 前端/UI | Next.js 14 + BFF + SSE/WebSocket + 图表体系 | 必须存在正式入口、正式页面、正式鉴权、正式测试 |
| 后端/API | FastAPI + Service/Repository/Worker + Trace + RBAC | 必须具备真实 API、真实审计、真实权限边界 |
| LLM/推理 | vLLM + Ollama + Triton + 多模型路由 | 必须具备 real-first 请求链路和降级链路 |
| 检索/RAG | Qdrant + OpenSearch/Elasticsearch + Rerank + GraphRAG | 必须具备真实搜索后端、真实图底座或等价真实图索引 |
| 集成 | OMS/WMS/SCM/CRM/FMS/BI/PaaS | 必须具备真实联调证据，不接受仅接口样板 |
| 数据平台 | Kafka + Schema + DLQ + Replay + Iceberg/Hudi + Flink/Spark/Ray | 必须具备真实数据流转和资产沉淀 |
| 运维/网关 | Kong + Prometheus/Grafana + EFK + K8s + HA | 必须具备真实部署和环境接入证据 |
| 安全 | SSO/OAuth2、RBAC、多租户、审计、数据脱敏、Prompt 防护 | 必须具备实际控制点和验证证据 |

## 3.2 可按“等价能力”验收的项目

以下项目可按“能力等价优先”验收，但必须书面说明取舍原因：

| 设计方案项 | 等价验收原则 |
|---|---|
| Vue3 + Element Plus | 若继续以 Next.js/React 为正式主栈，只要后台管理能力、组件成熟度、权限与运营场景完整，可按等价能力通过 |
| Milvus 辅助库 | 若 Qdrant 已承接主链路且无明确十亿级离线检索需求，可暂不作为前置阻塞项 |
| Dify / Langflow / CrewAI | 若现有 Agent 编排已满足真实企业场景，不强制要求全部框架落地 |
| Kettle / Airflow / Prefect 全量并存 | 允许按当前阶段引入最小可用调度组合，但必须满足实时、定时、事件驱动三类能力需求 |
