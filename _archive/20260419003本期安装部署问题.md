## 2. 已定位并已处理的问题

## 2.1 ZooKeeper 健康检查命令不适配

### 现象

原 compose 中使用：

```yaml
echo ruok | nc localhost 2181 | grep imok
```

在当前镜像与配置下，`4lw` 命令未开放，导致健康检查不稳定或失败。

### 已处理

- 健康检查改为：

```yaml
echo srvr | nc localhost 2181 | grep Mode
```

- 同时补充：

```yaml
ZOOKEEPER_4LW_COMMANDS_WHITELIST: srvr,ruok,stat,mntr,conf,isro
```

### 当前结论

- 该问题已解决
- `pms-local-zookeeper` 可以进入 `running healthy`

---

## 2.2 Kafka / Connect 内部互连地址不统一

### 现象

Kafka、`kafka-init`、`kafka-connect` 之间原先引用不统一，存在使用宿主地址或旧服务名的情况，容易导致：

- 内部 bootstrap 地址错误
- Connect 无法正确连接 broker
- 初始化 topic 阶段失败

### 已处理

- 统一改为显式容器名互连：
  - `pms-local-zookeeper:2181`
  - `pms-local-kafka:29092`
- `kafka-connect` 增加：
  - `host.docker.internal:host-gateway`

### 当前结论

- 该类配置问题已解决
- 当前剩余问题不再是容器间地址配置错误

### 现象

  - 自动按 `PartitionCount=1`
  - 自动补 `cleanup.policy=compact`

### 当前结论

- 该问题已解决

### 9.2 直接错误

当前已记录到 `artifacts/ops/local_graph_rag_neo4j_acceptance.json` 的核心错误为：

- `docker compose ... pull neo4j` 已成功，原先的 Docker Hub 拉取超时已解除

- `docker inspect` 显示容器健康状态为 `unhealthy`，日志为 `timed out starting health check` / `Health check exceeded timeout (10s)`
- `docker compose ... up -d --force-recreate neo4j` 一次返回 `502 Bad Gateway for API route http://%2F%2F.%2Fpipe%2FdockerDesktopLinuxEngine/...`，另一次长时间卡在 `Recreate`

- 这不是代码逻辑缺陷
