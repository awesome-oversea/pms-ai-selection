# Phase 5~8 企业级优化任务清单

> 依据文档：`docs/企业级问题清单.md`
> 基础前提：Phase 1~4 已完成，系统已从“可演示 PoC”收敛为“可信单体原型”
> 目标：推动系统从“可信单体原型”演进为“可治理、可扩展、可审计、可交付”的企业级生产系统
> 说明：本清单不再聚焦基础修修补补，而聚焦平台化、治理化、规模化与企业交付能力建设

---

## 1. 阶段划分说明

- **Phase 5：企业安全与租户治理基线**
  - 目标：建立多租户、正式权限模型、持久化审计和全链路追踪基础
  - 对应问题：E01、E02、E03、E04
- **Phase 6：任务平台化与模型治理**
  - 目标：让任务执行从单体进程能力升级为平台能力，并让模型调用具备成本、配额、路由治理能力
  - 对应问题：E05、E08、E06
- **Phase 7：知识平台化与数据治理**
  - 目标：让知识库从“可检索”升级为“可持续运营与优化”的企业知识平台
  - 对应问题：E07、E13、E12
- **Phase 8：企业发布、运行与交付治理**
  - 目标：形成可发布、可运维、可审计、可交付的企业级运营体系
  - 对应问题：E09、E10、E11、E14、E15

---

## 2. Phase 5：企业安全与租户治理基线

> 阶段目标：建立企业级系统最核心的边界能力——租户、权限、审计、链路追踪。没有这一阶段，后续平台化扩展会造成高返工成本。

### T5.1 建立租户模型与租户上下文

- **对应问题**：E01
- **任务目标**：将 `tenant_id` 作为系统的一等公民引入数据模型、认证上下文、服务层和日志链路
- **主要工作**：
  1. 新增 `Tenant`、`TenantConfig`、`TenantQuota` 等基础模型
  2. 为用户、任务、知识库、报告、审计等核心表设计租户关联策略
  3. 扩展 JWT/认证上下文，透出 `tenant_id`
  4. 在服务层引入 `TenantContext`
  5. 明确平台级资源与租户级资源边界
- **建议交付物**：
  - 租户领域模型设计文档
  - 数据模型 ER 图（包含 tenant 关系）
  - `TenantContext` 代码实现
  - 租户上下文传递约定说明
- **主要风险**：
  - 旧表 retrofitting 成本高
  - 需要明确哪些表属于平台级共享资源
- **验收标准**：
  - [ ] 核心业务模型存在明确 `tenant_id` 设计
  - [ ] 请求进入服务层后可以拿到 `tenant_id`
  - [ ] 日志/审计/任务创建至少能看到租户维度
  - [ ] 存在一组租户上下文单元测试或集成测试

---

### T5.2 核心业务表租户化改造

- **对应问题**：E01
- **任务目标**：将租户边界落到真实数据库结构和查询逻辑中
- **主要工作**：
  1. 为 `SelectionTask`、`Document`、`Chunk`、`Report` 等核心表新增 `tenant_id`
  2. 建立相关索引，如 `(tenant_id, status, created_at)`
  3. Repository 层默认追加租户过滤
  4. Qdrant payload 中加入 `tenant_id`
  5. 防止跨租户读取、更新、删除
- **建议交付物**：
  - Alembic migration 脚本
  - Repository 租户过滤基线
  - Qdrant payload 规范更新
- **主要风险**：
  - 数据迁移顺序和默认值处理复杂
  - 老数据回填策略需定义
- **验收标准**：
  - [ ] 核心表落库结构已支持租户字段
  - [ ] 所有核心查询默认按租户过滤
  - [ ] 向量检索与删除支持租户过滤
  - [ ] 存在跨租户访问被拒绝的验证用例

---

### T5.3 正式 RBAC / ABAC 权限体系

- **对应问题**：E02
- **任务目标**：从 `is_superuser` 过渡到正式角色权限体系，并为后续企业功能提供统一授权框架
- **主要工作**：
  1. 定义角色：`platform_admin / tenant_admin / operator / analyst / viewer / auditor`
  2. 定义权限资源与动作矩阵
  3. 支持租户内角色与平台级角色并存
  4. 把权限控制推进到 Service 层而非只停留在 endpoint
  5. 建立资源所有者/租户管理员/平台管理员三级访问规则
- **建议交付物**：
  - 角色权限矩阵文档
  - 权限检查中间件/依赖库
  - 资源级授权示例实现
- **主要风险**：
  - 权限体系过度设计导致实现迟滞
- **验收标准**：
  - [ ] 已定义正式角色与权限矩阵
  - [ ] 核心资源支持资源级授权
  - [ ] 非法访问返回正确的 401/403
  - [ ] 至少有一组权限集成测试验证职责分离

---

### T5.4 审计日志持久化与检索中心

- **对应问题**：E03
- **任务目标**：将当前内存审计升级为企业可用的持久化审计中心
- **主要工作**：
  1. 设计 `audit_logs` 表与索引
  2. 记录 `tenant_id / user_id / request_id / resource / action / result / before / after`
  3. 审计登录、权限拒绝、任务操作、知识库操作、配置变更、报告导出等事件
  4. 提供按时间、用户、资源、租户查询能力
  5. 规划归档与留存策略
- **建议交付物**：
  - 审计字段规范文档
  - 审计存储实现
  - 审计查询 API
- **主要风险**：
  - 审计记录量增长带来的存储压力
- **验收标准**：
  - [ ] 审计日志持久化存储
  - [ ] 重启后审计记录不丢失
  - [ ] 可按用户/资源/租户查询审计记录
  - [ ] 关键安全事件均有可追踪记录

---

### T5.5 request_id / trace_id 全链路追踪基线

- **对应问题**：E04
- **任务目标**：建立企业排障、SLA 统计和审计联动所需的链路关联能力
- **主要工作**：
  1. API 中间件生成 `request_id`
  2. 日志框架自动挂载 `request_id`
  3. 审计、任务、模型调用、知识库操作透传 `request_id`
  4. 接入 OpenTelemetry trace/span 基线
  5. 为外部依赖调用增加 trace 透传点
- **建议交付物**：
  - 链路追踪规范文档
  - Request/Trace 中间件
  - 日志/审计追踪集成代码
- **验收标准**：
  - [ ] 每个请求都有唯一 `request_id`
  - [ ] 日志、审计、任务均可按 `request_id` 串联
  - [ ] 至少一条关键链路可在日志中完整追踪

## 3. Phase 6：任务平台化与模型治理

> 阶段目标：把任务执行与模型调用从“代码功能”升级为“平台能力”，解决扩展性、稳定性、成本与隔离问题。

### T6.1 Worker 化执行架构落地

- **对应问题**：E05
- **任务目标**：实现 API 与长任务执行的物理/逻辑解耦
- **主要工作**：
  1. 定义任务队列模型
  2. 建立 Worker 进程/服务
  3. 任务领取、锁定、续租、心跳机制
  4. API 改为提交任务而非直接执行
- **交付物**：
  - Worker 架构设计
  - Worker 运行器
  - API/Worker 协议文档
- **验收标准**：
  - [ ] API 进程不直接执行长任务
  - [ ] Worker 可独立启动与扩缩容
  - [ ] 任务状态在 Worker 执行期间可持续观察

---

### T6.2 任务重试、超时、死信与优先级治理

- **对应问题**：E05
- **任务目标**：为任务平台建立企业级调度治理规则
- **主要工作**：
  1. 重试策略（次数、退避、可重试错误分类）
  2. 超时与取消一致性语义
  3. 死信队列/死信表
  4. 优先级与租户队列隔离
- **验收标准**：
  - [ ] 存在明确的重试与超时策略
  - [ ] 死信任务可查询与再处理
  - [ ] 高优先级任务与低优先级任务可区分执行

---

### T6.3 任务资源隔离与并发治理

- **对应问题**：E05、E08
- **任务目标**：防止任务洪峰冲垮模型、数据库和 API 进程
- **主要工作**：
  1. 任务并发上限
  2. 按租户/任务类型限流
  3. 资源配额管理
  4. Worker 水位和堆积告警
- **验收标准**：
  - [ ] 支持 per-tenant / per-task-type 并发治理
  - [ ] 任务洪峰不会明显影响 API 可用性
  - [ ] 存在队列堆积监控与告警

---

### T6.4 模型网关预算与配额治理

- **对应问题**：E08
- **任务目标**：建立租户级、模型级的成本和额度控制能力
- **主要工作**：
  1. 统计 token、调用量、成本
  2. 设定租户预算与配额
  3. 达到阈值时告警、降级或拒绝
  4. 模型调用计费归属到租户和功能域
- **验收标准**：
  - [ ] 每次模型调用都有成本记录
  - [ ] 每租户存在预算与配额规则
  - [ ] 超额度策略有明确行为

---

### T6.5 Prompt / Route Policy / 灰度治理

- **对应问题**：E08、E06
- **任务目标**：建立模型调用策略的可配置、可灰度、可回滚能力
- **主要工作**：
  1. Prompt 模板注册与版本化
  2. Route policy 外置配置
  3. Prompt 和 policy 的灰度发布
  4. A/B 测试与效果对比
- **验收标准**：
  - [ ] Prompt 有版本管理
  - [ ] 路由策略可配置且可回滚
  - [ ] 至少一项策略支持灰度验证

---

### T6.6 统一 API 契约与错误码体系

- **对应问题**：E06
- **任务目标**：为平台化扩展建立稳定 API 协议
- **主要工作**：
  1. 定义统一响应 envelope
  2. 定义错误码分层：认证、权限、业务、依赖、系统
  3. 统一 OpenAPI 与 SDK 生成基线
  4. 网关、BFF、前端统一消费协议
- **验收标准**：
  - [ ] 核心 API 全部使用统一 envelope
  - [ ] 错误码可被前端和集成方稳定消费
  - [ ] OpenAPI 文档与真实接口一致

---

## 4. Phase 7：知识平台化与数据治理

> 阶段目标：把已可用的知识库升级为企业知识平台，把业务数据升级为可治理数据资产。

### T7.1 文档版本化与索引重建治理

- **对应问题**：E07
- **任务目标**：让文档更新、回滚、重建索引都有正式机制
- **主要工作**：
  1. 文档版本号设计
  2. 支持增量更新与全量重建
  3. 旧索引保留/切换/回滚策略
  4. 版本间差异与状态追踪
- **验收标准**：
  - [ ] 文档支持版本化管理
  - [ ] 支持索引重建且状态可观察
  - [ ] 存在版本切换或回滚策略

---

### T7.2 真混合检索（Vector + BM25/ES）

- **对应问题**：E07
- **任务目标**：从当前 Qdrant/DB 回退检索升级为正式混合检索体系
- **主要工作**：
  1. 引入 ES / OpenSearch / BM25 真实召回层
  2. 向量召回与关键词召回融合
  3. 支持 tenant/knowledge_base/document 过滤
  4. 调整检索结果归一化策略
- **验收标准**：
  - [ ] 向量与关键词召回同时可用
  - [ ] 检索融合策略有可配置参数
  - [ ] 至少有一组混合检索效果验证案例

---

### T7.3 Rerank 精排与引用质量治理

- **对应问题**：E07
- **任务目标**：提升引用质量与可解释性
- **主要工作**：
  1. 接入 rerank model
  2. 检索结果精排
  3. 返回更稳定的引用片段和排序理由
  4. 质量指标：召回、精排、一致性、引用正确率
- **验收标准**：
  - [ ] Rerank 可被真实调用
  - [ ] 检索结果包含引用片段和排序元数据
  - [ ] 至少一项质量指标可持续观测

---

### T7.4 知识库质量评估与运营看板

- **对应问题**：E07、E09
- **任务目标**：使知识平台具备持续优化能力
- **主要工作**：
  1. 建立标准评测集
  2. 指标：hit@k、MRR、引用正确率、索引失败率
  3. 看板展示与告警
  4. 文档/知识库健康评分
- **验收标准**：
  - [ ] 至少一套 RAG 评测集可运行
  - [ ] 索引与检索质量指标可观测
  - [ ] 存在质量退化告警机制

---

### T7.5 统一主数据与业务数据域模型

- **对应问题**：E13
- **任务目标**：建立 Product / Trend / Recommendation / ERP 之间的一致性数据模型
- **主要工作**：
  1. 定义核心主数据实体与口径
  2. 定义 Source of Truth
  3. 规划数据同步方向和 CDC/ETL 策略
  4. 建立数据字典与字段血缘说明
- **验收标准**：
  - [ ] 核心实体有统一主模型定义
  - [ ] 数据口径与来源清晰
  - [ ] 至少有一份字段字典和数据血缘说明

---

### T7.6 配置中心与特性开关基线

- **对应问题**：E12
- **任务目标**：支持租户差异化配置、灰度开关和策略动态调整
- **主要工作**：
  1. 配置中心建模
  2. Feature Flag 体系
  3. 支持 tenant / env / version 级覆盖
  4. 配置变更审计与回滚
- **验收标准**：
  - [ ] 功能开关不依赖重新发版
  - [ ] 配置变更可审计、可回滚
  - [ ] 至少支持一种租户差异化配置

---

## 5. Phase 8：企业发布、运行与交付治理

> 阶段目标：把系统从“研发可运行”提升到“企业可发布、可运维、可交付”。

### T8.1 技术/业务/经营三层指标体系

- **对应问题**：E09
- **任务目标**：建立运维、产品、管理统一可用的观测体系
- **主要工作**：
  1. 技术指标：API、DB、Redis、Qdrant、LLM、Worker
  2. 业务指标：任务成功率、审批 SLA、索引成功率、检索命中率
  3. 经营指标：租户调用量、token 消耗、成本、任务量
- **验收标准**：
  - [ ] 三类指标均有数据来源和展示方式
  - [ ] 至少有一套 Prometheus/Grafana 看板
  - [ ] 核心异常存在告警规则

---

### T8.2 企业发布质量门禁

- **对应问题**：E10
- **任务目标**：建立正式发布前的质量门槛
- **主要工作**：
  1. 单元/契约/集成/E2E/性能/安全测试分层
  2. 数据迁移 smoke test
  3. 发布前 smoke checklist
  4. 质量报告与阻断策略
- **验收标准**：
  - [ ] 发布前至少经过契约、集成、迁移、smoke 验证
  - [ ] 核心测试失败会阻止发布
  - [ ] 本地可复现核心门禁

---

### T8.3 网关/BFF/开放接口分层治理

- **对应问题**：E11、E06
- **任务目标**：建立内外部接口边界与流量治理能力
- **主要工作**：
  1. API Gateway 外置认证、限流、版本控制
  2. 前端 BFF 聚合接口
  3. 开放平台接口规范与 SDK 输出
  4. 接口兼容与版本策略
- **验收标准**：
  - [ ] 内部接口、前端接口、外部开放接口边界清晰
  - [ ] 存在版本管理与限流策略
  - [ ] 至少一套外部接口可由 OpenAPI/SDK 描述

---

### T8.4 Alembic / 迁移治理 / 回滚策略

- **对应问题**：E14
- **任务目标**：建立企业数据库变更治理体系
- **主要工作**：
  1. 引入 Alembic 正式迁移基线
  2. 定义 migration review 规范
  3. 发布与回滚策略
  4. schema 前后兼容检查
- **验收标准**：
  - [ ] 所有结构变更通过 migration 管理
  - [ ] 存在回滚流程与验证方法
  - [ ] 生产发布不再依赖 `create_all`

---

### T8.5 Runbook / Oncall / SLA / 变更流程文档

- **对应问题**：E15
- **任务目标**：形成可交付、可交接、可值班的企业运行体系
- **主要工作**：
  1. 编写启动/停机/排障 Runbook
  2. 编写值班与事故升级手册
  3. 定义 SLA/SLO 与告警级别
  4. 定义权限申请、配置变更、索引重建、发布审批流程
- **验收标准**：
  - [ ] 存在标准运行手册
  - [ ] 故障、变更、发布流程有明确文档
  - [ ] 非核心研发也能按文档完成基础运维动作

## 8. 排期与职责建议

### 8.1 建议团队角色配置

- **产品负责人 / 业务架构师**：负责多租户边界、角色体系、业务口径与验收标准
- **后端架构师 / 平台工程师**：负责租户模型、Worker 化、统一 API 契约、迁移治理
- **AI 平台工程师**：负责模型网关治理、Prompt 管理、RAG 平台化、检索评估
- **数据工程师**：负责主数据模型、数据治理、指标与数仓口径
- **SRE / DevOps**：负责 CI/CD、发布治理、可观测性、Runbook 与告警体系
- **安全 / 合规负责人**：负责 RBAC、审计、权限模型、日志留存与审查要求
- **QA / 测试负责人**：负责企业发布门禁、迁移测试、性能/安全测试体系

### 8.2 建议排期节奏

- **Phase 5**：2~4 个 Sprint
- **Phase 6**：2~4 个 Sprint
- **Phase 7**：2~4 个 Sprint
- **Phase 8**：2~3 个 Sprint

> 建议采用“每阶段先做最小闭环、再做增强项”的策略，不建议一口气并行启动所有主题，否则会再次形成大蓝图先行、真实交付滞后的风险。

### 8.3 建议里程碑

- **M1：企业边界建立**
  - 完成租户、权限、审计、追踪四件套
- **M2：平台执行能力建立**
  - 完成 Worker 化、任务治理、模型治理
- **M3：知识与数据平台建立**
  - 完成知识平台治理和主数据治理基线
- **M4：生产发布与交付就绪**
  - 完成发布门禁、运行手册、SLA、迁移治理

## 9. 实施建议

- 建议每个阶段先做“架构最小闭环”，再做“能力扩展”，避免再次进入“大蓝图先行、代码证据滞后”的状态
- 建议每个任务都要求：
  1. 明确前置依赖
  2. 明确影响面
  3. 明确验收环境
  4. 明确 real-pass / mock-pass / skipped 口径
- 建议从 Phase 5 开始同步建立企业级阶段验收清单，避免等到代码做完再补验收口径
