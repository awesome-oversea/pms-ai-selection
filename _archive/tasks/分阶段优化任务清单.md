# 分阶段优化任务清单

> 依据文档：`docs/核心问题清单.md`
> 目标：围绕核心问题分阶段推进整改，确保每个任务都有明确验收标准，便于逐步实施与验收
> 说明：本清单优先服务“当前项目可信收敛”，不是继续扩张目标蓝图

---

## 1. 阶段划分说明

- **Phase 1：基线收口**
  - 目标：统一口径、依赖、启动、探针、部署基线
  - 对应问题：K01、K02、K07
- **Phase 2：主链路可信化**
  - 目标：让选品任务链路从“能演示”变成“可追踪、可恢复、可验收”
  - 对应问题：K03、K08
- **Phase 3：知识库与 Provider 真闭环**
  - 目标：让知识库、LLM、外部能力边界清晰且可验证
  - 对应问题：K05、K06
- **Phase 4：架构收敛与质量门禁**
  - 目标：解耦 Web/API，建立可信测试和基本权限边界
  - 对应问题：K04、K09、K10

---

## 2. Phase 1：基线收口

> 阶段目标：形成一致的“当前实现口径 + 可运行基线 + 可部署基线”

### T1.1 当前能力边界文档收口

- **对应问题**：K01
- **任务目标**：将“当前已实现能力”和“目标架构能力”分开描述，修正文档口径漂移
- **主要工作**：
  1. 补充当前系统能力说明文档
  2. 修订 README 中与现状不符的能力描述
  3. 对方案文档增加“目标态/规划态”标识
  4. 明确哪些模块为 mock、stub、规划中、已真实接入

**验收标准**：
- [ ] 存在单独的“当前系统能力边界”文档
- [ ] README 不再把未落地能力描述为已完成
- [ ] 方案文档中所有未落地模块均有“目标态/规划态”标识
- [ ] 文档中至少包含一张“已实现 / 部分实现 / 未实现”能力矩阵

---

### T1.2 验收口径与问题基线统一

- **对应问题**：K01
- **任务目标**：统一任务、问题、验收三套文档的判断口径
- **主要工作**：
  1. 建立问题 ID 与任务 ID 的映射关系
  2. 建立阶段验收清单
  3. 统一测试通过数和验收结论的表述方式
  4. 将“mock 通过”与“真实环境通过”分开记录

**验收标准**：
- [ ] 每个阶段任务可追溯到至少一个问题 ID
- [ ] 存在统一的阶段验收清单文档
- [ ] 测试报告中区分 mock / real / skipped 三类结果
- [ ] 不再出现多个文档对同一阶段给出互相冲突的完成结论

---

### T1.3 依赖源统一与安装基线修复

- **对应问题**：K07
- **任务目标**：统一依赖声明，消除安装漂移
- **主要工作**：
  1. 选定 `pyproject.toml` 作为单一依赖源
  2. 清理 `requirements.txt` 中与真实代码不一致的依赖
  3. 补齐 Kafka、Qdrant、Redis、测试等真实使用依赖
  4. 形成标准安装命令和锁定方式

**验收标准**：
- [ ] 依赖管理以单一文件为准，不存在双源冲突
- [ ] 新环境执行标准安装命令后无缺失依赖错误
- [ ] `requirements.txt` 如保留，必须由标准方式导出而非手工维护
- [ ] Kafka、Qdrant、Redis 相关导入均能在安装后通过基础 import 验证

---

### T1.4 应用生命周期与初始化逻辑修复

- **对应问题**：K02、K07
- **任务目标**：统一应用启动/关闭行为，明确关键依赖的失败策略
- **主要工作**：
  1. 收敛 DB、Redis、Qdrant 初始化逻辑
  2. 明确 fail fast 与 warning-only 场景
  3. 关闭阶段补齐资源释放
  4. 启动日志输出依赖状态摘要

**验收标准**：
- [ ] 应用启动日志能显示数据库、Redis、Qdrant 初始化结果
- [ ] 关键依赖失败时应用行为符合预设策略，并有明确日志
- [ ] 应用关闭时数据库连接池、Redis、Qdrant 客户端被正常释放
- [ ] 连续两次启动/关闭不出现资源占用或重复初始化异常

---

### T1.5 健康探针与部署基线修复

- **对应问题**：K02
- **任务目标**：保证本地、Docker、K8s 的健康探针与应用真实路由一致
- **主要工作**：
  1. 统一 `/health`、`/ready`、`/live` 路由
  2. 修复 K8s readiness/liveness probe 路径
  3. 修正 readiness 状态码语义
  4. 输出最小部署说明

**验收标准**：
- [ ] 应用存在可访问的 `/health`、`/ready`、`/live` 接口，语义清晰
- [ ] 当数据库或 Redis 不可用时，`/ready` 返回非 2xx
- [ ] K8s 探针路径与应用真实路径一致
- [ ] 本地启动、容器启动、K8s 部署三种方式下的健康检查结果一致

---

### T1.6 Dockerfile 与 Compose 最小可运行化

- **对应问题**：K02
- **任务目标**：从“伪高可用演示配置”收敛到“真实可运行开发配置”
- **主要工作**：
  1. 修复 Dockerfile 中不合法或不可移植写法
  2. 清理 Compose 中无法真实工作的主从/哨兵伪配置
  3. 提供最小开发依赖编排：App + DB + Redis + 可选 Qdrant
  4. 验证容器间启动顺序和环境变量基线

**验收标准**：
- [ ] `docker build` 能成功构建镜像
- [ ] `docker compose up` 能拉起最小可用开发环境
- [ ] 应用容器启动后健康检查可通过
- [ ] Compose 文件中不再出现无法真实工作的“伪高可用”说明性配置

## 3. Phase 2：主链路可信化

> 阶段目标：让任务创建、执行、查询、取消、结果获取形成持久化闭环

### T2.1 建立任务 Repository 与 Service 基线

- **对应问题**：K03、K08
- **任务目标**：将选品任务数据访问和业务编排从 endpoint 中拆出
- **主要工作**：
  1. 新增任务 Repository
  2. 新增任务 Application Service
  3. 统一任务创建、状态更新、结果读取接口
  4. 统一任务对象的输入输出 schema

**验收标准**：
- [ ] 存在独立的任务 Repository 与 Service 模块
- [ ] endpoint 不再直接承担数据库 CRUD 细节
- [ ] 任务创建、查询、状态更新均通过 Service 完成
- [ ] 任务 schema 在项目中只有一套主定义并被 API 统一使用

---

### T2.2 去除选品接口的核心内存态依赖

- **对应问题**：K03
- **任务目标**：消除 `_task_store`、`_master_store`、`_approval_store` 对主链路的支撑
- **主要工作**：
  1. 将任务元数据落库
  2. 将审批状态落库或纳入统一状态表
  3. 将任务执行阶段和结果摘要落库
  4. 清理 endpoint 对全局字典的直接访问

**验收标准**：
- [ ] `selection.py` 中不再存在主流程依赖的全局内存字典
- [ ] 创建任务后数据库中能看到任务记录
- [ ] 服务重启后任务详情与历史结果仍可查询
- [ ] 取消任务操作更新数据库状态，而不是仅修改进程内对象

---

### T2.3 统一任务 ID 与工作流 Session ID

- **对应问题**：K03、K08
- **任务目标**：消除外部 task_id 与内部 session_id 断链问题
- **主要工作**：
  1. 统一 ID 传递规则
  2. 在 SelectionMaster 运行态中显式使用 task_id/session_id
  3. 日志、结果、数据库记录均引用同一主键链路

**验收标准**：
- [ ] API 返回的 task_id 能在日志、数据库、工作流状态中唯一对应
- [ ] 不存在“API task_id 与内部 session_id 不一致”的情况
- [ ] 任意任务执行记录可通过 task_id 完整追踪到阶段状态与结果

---

### T2.4 建立任务状态机与取消语义

- **对应问题**：K03
- **任务目标**：明确任务生命周期和状态流转规则
- **主要工作**：
  1. 定义任务状态枚举和允许的流转
  2. 为取消、失败、完成设置明确语义
  3. 实现基础取消控制点
  4. 为后续 Worker 化保留状态兼容性

**验收标准**：
- [ ] 任务状态定义至少覆盖 `pending/running/completed/failed/cancelled`
- [ ] 非法状态流转会被拒绝并返回明确错误
- [ ] 执行中的任务可被取消，并在最终状态中体现为 `cancelled`
- [ ] 状态变更有时间戳和原因记录

---

### T2.5 统一任务结果结构与关键字段语义

- **对应问题**：K08
- **任务目标**：统一任务结果、统计字段、报告字段的结构定义
- **主要工作**：
  1. 统一 `go_no_go` 结构
  2. 统一报告内容字段类型
  3. 清理 endpoint 层重复 schema
  4. 确保 OpenAPI 与真实响应一致

**验收标准**：
- [ ] 关键结果字段仅保留一套对外 schema
- [ ] OpenAPI 示例与真实接口返回结构一致
- [ ] 统计接口读取的字段结构与任务结果落库结构一致
- [ ] 前端页面或 API 调试不再出现同字段多种结构的情况

---

### T2.6 预留 Worker 化改造接口

- **对应问题**：K03
- **任务目标**：先不一次性上消息队列，但要为异步任务迁移留好边界
- **主要工作**：
  1. 抽象任务提交入口
  2. 抽象执行器接口
  3. 将 `BackgroundTasks` 封装在适配层，不再散落在 endpoint

**验收标准**：
- [ ] endpoint 层不直接包含复杂的后台执行编排
- [ ] 存在可替换的任务执行器接口
- [ ] 未来接入 Celery/RQ/Arq/自研 Worker 时不需要重写 API 契约

---

## 4. Phase 3：知识库与 Provider 真闭环

> 阶段目标：让知识库真的可用，让 mock 和 real 清晰分层

### T3.1 知识库文档持久化基线

- **对应问题**：K05
- **任务目标**：将知识库文档、切片与状态写入持久化存储
- **主要工作**：
  1. 文档元信息落库
  2. chunk 数据落库
  3. 建立索引状态字段
  4. 错误状态记录失败原因

**验收标准**：
- [ ] 上传文档后数据库中存在 Document 记录
- [ ] 文档切片后数据库中存在 Chunk 记录
- [ ] 文档状态至少覆盖 `pending/processing/indexed/failed`
- [ ] 上传失败时能记录失败原因并通过接口返回

---

### T3.2 打通 embedding 与 Qdrant 入库链路

- **对应问题**：K05、K06
- **任务目标**：让上传文档真正进入向量检索体系
- **主要工作**：
  1. 接入 embedding provider
  2. 将 chunk 向量写入 Qdrant
  3. 建立文档与向量点位的映射关系
  4. 处理重建索引与重复导入

**验收标准**：
- [ ] 上传后的文档 chunk 能生成向量并写入 Qdrant
- [ ] Qdrant 中可按文档或知识库维度查询到对应点位
- [ ] 同一文档重复入库时有明确幂等策略
- [ ] 索引失败时文档状态变为 `failed`，成功时变为 `indexed`

---

### T3.3 修复知识库查询链路

- **对应问题**：K05
- **任务目标**：让查询从真实持久化和向量库数据中检索，而不是依赖内存对象
- **主要工作**：
  1. 查询服务读取知识库和文档配置
  2. 检索器从真实索引恢复
  3. 返回结果中带引用来源与置信信息
  4. 支持按知识库范围过滤

**验收标准**：
- [ ] 上传文档后，查询接口可命中该文档内容
- [ ] 查询结果包含文档来源、chunk 或引用片段信息
- [ ] 重启服务后已上传文档仍可被检索到
- [ ] 查询不再依赖临时进程内 `_doc_store` 或空 retriever 初始化

---

### T3.4 Provider 分层：mock / real 显式化

- **对应问题**：K06
- **任务目标**：为 LLM、Embedding、ERP、GraphRAG 等外部能力建立统一 provider 分层
- **主要工作**：
  1. 定义 provider interface
  2. 明确 mock provider 和 real provider
  3. 配置层显式声明 provider_mode
  4. 日志和响应元数据中记录 provider_mode

**验收标准**：
- [ ] 至少 LLM 与 Embedding 两类 provider 已完成 mock/real 分层
- [ ] 配置中可显式选择运行模式
- [ ] 日志中能看到当前请求使用的 provider_mode
- [ ] mock 模式与 real 模式的行为边界清晰，不再混用实现细节

---

### T3.5 LLMGateway 真实调用闭环

- **对应问题**：K06、K07
- **任务目标**：让网关支持真实 HTTP 调用与降级逻辑
- **主要工作**：
  1. 接通 OpenAI 兼容或 vLLM 接口
  2. 接通降级 provider（如 Ollama）
  3. 保留 mock 但不默认伪装成 real
  4. 记录调用耗时、错误、熔断和降级路径

**验收标准**：
- [ ] `use_mock=false` 时能够向真实模型服务发起请求
- [ ] 网络错误或超时能触发明确降级逻辑
- [ ] 响应中能区分主 provider、降级 provider、mock provider
- [ ] 调用日志包含耗时、状态、错误摘要

---

## 5. Phase 4：架构收敛与质量门禁

> 阶段目标：降低耦合，建立最小可信测试集和基础权限边界

### T4.1 Web 与 API 解耦

- **对应问题**：K04
- **任务目标**：取消 Web 对 API 内部变量和注册表的直接依赖
- **主要工作**：
  1. 提供查询服务给 Web 层使用
  2. 清理 `routes.py` 中对 endpoint 内部对象的直接 import
  3. 移除导入副作用逻辑

**验收标准**：
- [ ] Web 路由不再直接 import endpoint 内部状态变量
- [ ] 页面展示数据全部来自服务层或正式 API
- [ ] 模块 import 不再自动创建目录等副作用
- [ ] Web 与 API 任一侧重构时不会直接破坏另一侧运行

---

### T4.2 重建最小可信测试集

- **对应问题**：K10
- **任务目标**：用少量但可信的测试覆盖核心主链路
- **主要工作**：
  1. 设计单元测试：任务状态机、服务层、schema 契约
  2. 设计集成测试：DB/Redis/选品 API/RAG 上传查询
  3. 清理与当前实现漂移严重的测试
  4. 输出统一测试分层说明

**验收标准**：
- [ ] 存在一组最小可信测试集，并能一键运行
- [ ] 至少覆盖：启动探针、任务创建查询、任务状态流转、RAG 上传检索
- [ ] 明显漂移或无效的旧测试已标记废弃或移除门禁
- [ ] 测试报告可区分单元、集成、E2E 三层

---

### T4.3 接入 CI 质量门禁

- **对应问题**：K10、K07
- **任务目标**：让核心测试和静态检查成为提交门禁
- **主要工作**：
  1. 配置 lint/type/test 基线
  2. 在 CI 中执行最小可信测试集
  3. 失败时输出明确原因

**验收标准**：
- [ ] 存在可执行的 CI 配置文件
- [ ] CI 至少执行依赖安装、静态检查、核心测试
- [ ] 任一核心测试失败时 CI 返回失败
- [ ] 团队能用同一命令在本地复现 CI 核心检查

---

### T4.4 基础 RBAC 与权限保护

- **对应问题**：K09
- **任务目标**：为关键接口补齐基础鉴权和角色边界
- **主要工作**：
  1. 定义基础角色，如 `admin`、`user`
  2. 对任务管理、知识库管理、报告查看等接口加角色保护
  3. 补齐未授权访问处理

**验收标准**：
- [ ] 关键管理接口需要认证后访问
- [ ] 不同角色访问受限资源时返回正确的 401/403
- [ ] 角色定义与接口保护规则有文档说明
- [ ] 有至少一组权限测试验证角色边界

---

### T4.5 审计日志基线

- **对应问题**：K09
- **任务目标**：对关键操作建立基础留痕能力
- **主要工作**：
  1. 审计登录、任务创建、任务取消、文档上传、权限拒绝等事件
  2. 记录操作人、时间、对象、结果
  3. 提供最小查询或导出能力

**验收标准**：
- [ ] 关键操作会生成审计记录
- [ ] 审计记录至少包含用户、操作、对象、时间、结果
- [ ] 可按任务或用户查询到对应审计记录
- [ ] 权限拒绝等安全相关事件也会被记录

---
