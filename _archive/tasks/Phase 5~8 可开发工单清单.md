# Phase 5~8 可开发工单清单

> 依据文档：`docs/企业级问题清单.md`、`tasks/Phase 5~8 企业级优化任务清单.md`
> 适用范围：企业级演进阶段的研发排期、Epic 拆解、Sprint 规划、研发与测试协同
> 说明：本清单将 Phase 5~8 进一步细化到可开发工单粒度，采用 `T5.1.1 / T5.1.2 / T5.1.3` 编码方式，强调“可实现、可验收、可排期、可协作”

---

## 1. 使用说明

### 1.1 编码规则

- `T5.x.x`：Phase 5 工单
- `T6.x.x`：Phase 6 工单
- `T7.x.x`：Phase 7 工单
- `T8.x.x`：Phase 8 工单

### 1.2 每个工单建议最少包含的信息

- 工单目标
- 优先级（P0/P1/P2）
- 影响范围
- 主要实现点
- 前置依赖
- 依赖类型（数据模型/中间件/平台能力/外部服务）
- 风险
- 建议 Owner 角色
- 预计工时（人天/人周）
- 验收方式

### 1.3 实施原则

- 优先做“架构最小闭环”，再做“增强能力”
- 不允许先写大蓝图、后补实现证据
- 每个工单必须能映射到企业级问题 ID
- 每个工单必须有明确 real-pass 验收口径

# 2. Phase 5：企业安全与租户治理基线

> 阶段目标：建立企业系统运行的最小安全边界与租户边界，为后续平台化和多客户接入提供基础设施。

## T5.1 建立租户模型与租户上下文

### T5.1.1 建立 Tenant 基础模型
- **对应问题**：E01
- **目标**：新增 `Tenant`、`TenantConfig`、`TenantQuota` 基础实体
- **实现点**：
  - 建模租户基本信息、状态、套餐、配额
  - 支持启用/禁用/试用/正式租户状态
  - 预留计费、隔离策略、默认 provider 配置字段
- **前置依赖**：无
- **风险**：平台级资源与租户级资源边界定义不清
- **验收方式**：
  - 数据库存在 Tenant 相关模型与 migration
  - 至少支持创建、查询、启停租户

### T5.1.2 JWT 与认证上下文透出 tenant_id
- **对应问题**：E01、E02
- **目标**：让认证上下文自带 `tenant_id`
- **实现点**：
  - JWT payload 增加 `tenant_id`
  - `get_current_user()` 返回 tenant 信息
  - 匿名、平台级、租户级用户上下文语义区分
- **前置依赖**：T5.1.1
- **验收方式**：
  - 登录后 token 中存在 tenant_id
  - 服务层可统一获取 tenant context

### T5.1.3 建立 TenantContext 抽象
- **对应问题**：E01、E04
- **目标**：统一服务层/仓储层使用的租户上下文对象
- **实现点**：
  - 定义 `TenantContext`
  - 将其注入 service / repository / audit / metrics
  - 避免 tenant_id 在代码中分散透传
- **前置依赖**：T5.1.2
- **验收方式**：
  - 核心 service 可直接消费 TenantContext
  - 至少一条业务链路无手写 tenant_id 透传

### T5.1.4 平台级资源与租户级资源边界建模
- **对应问题**：E01
- **目标**：明确哪些资源共享、哪些资源隔离
- **实现点**：
  - 输出资源分层矩阵：平台共享 / 租户专属 / 租户可继承
  - 明确模型策略、知识库、审计、指标、报表的边界
- **前置依赖**：T5.1.1
- **验收方式**：
  - 存在一份资源边界矩阵文档
  - 后续实现不再反复争议资源归属

## T5.2 核心业务表租户化改造

### T5.2.1 SelectionTask / AgentRun / SelectionResult 加 tenant_id
- **对应问题**：E01
- **目标**：任务域数据正式租户化
- **实现点**：
  - 增加字段、索引、回填策略
  - 查询默认按 tenant 过滤
- **前置依赖**：T5.1.3
- **验收方式**：
  - 核心任务表都具备 tenant_id
  - 跨租户读取失败

### T5.2.2 Document / Chunk / KnowledgeBase 租户化
- **对应问题**：E01、E07
- **目标**：知识域数据正式租户化
- **实现点**：
  - `KnowledgeBase / Document / Chunk` 新增 tenant_id
  - 默认知识库变为租户默认知识库
- **前置依赖**：T5.1.3
- **验收方式**：
  - 不同租户文档与 chunk 不串读
  - 默认知识库不跨租户共享

### T5.2.3 Report / Audit / CostUsage 租户化
- **对应问题**：E01、E03、E08
- **目标**：报表、审计、成本统计具备租户归属
- **实现点**：
  - 审计表、成本表、报告表补 tenant_id
  - 支持租户维度汇总
- **前置依赖**：T5.1.3
- **验收方式**：
  - 所有审计、报表、成本事件可按租户查询

### T5.2.4 Repository 默认租户过滤基线
- **对应问题**：E01
- **目标**：避免租户过滤全靠人工记忆
- **实现点**：
  - 建立 BaseRepository 或统一 filter helper
  - 默认查询必须带 tenant_id
- **前置依赖**：T5.2.1 ~ T5.2.3
- **验收方式**：
  - 核心 repository 无 tenant 的查询会被显式拒绝或标注平台级例外

### T5.2.5 Qdrant payload 增加 tenant_id 过滤
- **对应问题**：E01、E07
- **目标**：向量检索也具备租户边界
- **实现点**：
  - upload / query / delete 全链路带 tenant_id
  - Qdrant filter 强制 tenant 过滤
- **前置依赖**：T5.2.2
- **验收方式**：
  - 跨租户检索不会返回他租户文档

## T5.3 正式 RBAC / ABAC 权限体系

### T5.3.1 定义角色矩阵
- **对应问题**：E02
- **目标**：确定平台级角色与租户级角色
- **实现点**：
  - 定义角色、职责、可访问资源
  - 输出角色矩阵
- **验收方式**：
  - 存在正式角色矩阵文档

### T5.3.2 定义权限资源与动作模型
- **对应问题**：E02
- **目标**：建立 `resource + action` 模型
- **实现点**：
  - 权限标识规范
  - 资源动作清单
- **验收方式**：
  - 核心资源都能映射到权限动作

### T5.3.3 权限检查从 endpoint 下沉到 service
- **对应问题**：E02
- **目标**：避免仅靠路由依赖做权限控制
- **实现点**：
  - service 层资源级权限检查
  - 所有者/租户管理员/平台管理员多级决策
- **验收方式**：
  - 资源访问越界会在 service 层被拒绝

### T5.3.4 增加 ABAC 条件授权
- **对应问题**：E02
- **目标**：支持资源状态、租户、所有者等条件控制
- **实现点**：
  - 基于属性的授权规则引擎最小实现
- **验收方式**：
  - 至少一个资源支持状态条件授权

### T5.3.5 权限管理接口与权限测试集
- **对应问题**：E02、E10
- **目标**：权限可配置、可验证
- **实现点**：
  - 角色分配接口
  - 权限校验测试用例
- **验收方式**：
  - 角色分配、权限变更有测试覆盖
