## 1. 方案定位

本方案不是重新定义终极架构，而是在既有设计方案基础上，给出当前项目的**下一阶段可执行落地方案**。

当前阶段的核心任务不是继续横向增加模块，而是把已经铺开的架构能力收口成一条**真实可运行、可验证、可交付**的业务主链路。

本阶段建议命名为：

**Phase-N：真实闭环收口阶段**

## 2. 当前阶段判断

结合现有代码、测试和架构文档，项目现状可以概括为：

1. 分层架构、业务闭环、Agent 编排、知识库、工作台、治理能力已经基本成型。
2. 大量能力处于 `mock / compatible / local-real / remote-service` 混合状态。
3. 真实外部联调、真实推理、真实数据回流、真实基础设施运行证据仍然不足。
4. 当前项目最缺的不是新模块，而是“真实黄金链路”与“真实运行证据”。

因此，下一步项目方案必须围绕以下主目标展开：

- 做透一条真实黄金链路
- 固化模式治理
- 收口真实依赖
- 补齐运行级测试证据
- 为后续 K8s / HA / 多模态 / GraphRAG 升级建立稳定基线

## 3. 本阶段目标

### 3.1 总体目标

在 6 周内，完成一条从“选品任务创建”到“执行反馈回流”的最小真实业务闭环，并形成稳定验收包。

### 3.2 业务目标

以“**户外储能电源**”作为标准样板场景，打通以下链路：

1. 创建选品任务
2. 触发 Agent 编排执行
3. 输出推荐结果与风险结论
4. 完成审批流
5. 采纳推荐并生成执行动作
6. 接入 OMS / CRM / FMS / BI 的最小真实回流
7. 完成再评分与报告导出
8. 在工作台、日志、指标、审计中可追踪

### 3.3 技术目标

- 默认把关键链路从 `mock` 收口到 `local-real` 或 `remote-service`
- 固化单节点基础设施闭环：Postgres、Redis、Qdrant、OpenSearch、Kafka、Kong
- 修复可信回归中的阻塞失败项
- 建立“结构验证 / 本地运行验证 / 真实依赖验证”三层测试体系

### 3.4 验收目标

本阶段结束时，至少满足以下标准：

- 黄金链路可以在本地或准生产环境重复执行
- 工作台可以真实展示主链路结果而不是仅展示静态蓝图
- 关键状态接口能够明确返回当前模式、fallback 和依赖健康度
- 可信回归通过率恢复到高可用基线，黄金链路无阻塞失败
- 输出完整验收证据：运行截图、接口日志、测试报告、状态快照

## 4. 方案原则

### 4.1 收口优先

不再以“页面数量”“模块数量”“接口数量”作为主要推进指标，改为以主链路闭环完整度作为核心指标。

### 4.2 真实优先

默认优先推进：

- `local-real`
- `remote-service`

非必要不继续扩展新的 `mock-only` 能力。

### 4.3 分层不推翻

保留现有分层架构：

- 用户接入层
- 网关层
- UI 层
- API 层
- Agent 编排层
- AI 中台
- LLM / 推理引擎层
- 数据处理层
- 数据存储层
- 内外部集成层
- 监控运维层
- 基础设施层

本阶段重点是把这些层之间的关键依赖收口，而不是重做架构。

### 4.4 证据优先

任何“完成”必须同时具备：

- 代码落点
- 运行证据
- 测试证据
- 状态可观测性

## 6. 核心实施策略

### 6.1 业务策略：围绕黄金链路建立样板

选定一个标准任务样板：

- 品类：户外储能电源
- 目标市场：美国
- 数据输入：Amazon + Google Trends + 内部 OMS / CRM / FMS / BI 最小集
- 输出：推荐、风险、利润、审批、执行反馈、报告

这样做的目的不是限制业务，而是先建立一个可复制的标准样板，后续再扩到更多品类。

### 6.2 编排策略：当前阶段接受 compatible，下一阶段再切原生

基于现有代码，当前编排核心仍以 `LangGraphCompatibleRunner` 为主，且已经具备：

- 图结构描述
- 并行执行
- 断点 / 快照 / 恢复
- 风险与报告串联

因此本阶段策略是：

- **先把现有 compatible 路径做成正式运行基线**
- 原生 LangGraph runtime 迁移放到下一阶段作为专项

### 6.3 AI 策略：先主链路真实化，再扩高级能力

本阶段优先级：

1. LLM Gateway 主路径去 `mock`
2. RAG 检索和写入稳定
3. 报告生成稳定
4. 再考虑多模态、微调、GraphRAG 深化

### 6.4 数据流策略：先单节点跑通，再升级分布式

本阶段采用：

- Kafka 单节点
- Postgres 单节点
- Redis 单节点
- Qdrant / OpenSearch 单节点
- Outbox / CDC compatible 路径

先保证链路稳定，再进入 Flink、Iceberg、K8s、HA 的升级阶段。

### 6.5 前端策略：从蓝图验收到真实交互验收

前端本阶段不再以页面数量为主，而是要把以下真实交互做实：

- 登录与鉴权
- 创建任务
- 查看执行流
- 审批
- 反馈
- 报告下载
- 工作台状态刷新

## 7. 工作流分解

- 重复执行结果结构一致
- 审批、采纳、反馈动作在 API 与前端都可验证

### 工作流 B：模式治理与运行口径收口

目标：

- 明确当前所有核心能力的运行模式和降级规则

交付物：

- 模式矩阵文档

验收：

- `mock / compatible / local-real / remote-service` 口径统一
- 状态页能看出当前实际运行模式

### 工作流 C：真实依赖与基础设施基线

目标：

- 单节点基础设施一键启动、稳定可观测

交付物：

- 本地栈启动脚本
- 健康检查脚本
- Kong upstream 校验
- Kafka / OpenSearch / Qdrant / Redis / Postgres 状态页

验收：

- 一键拉起成功
- 服务健康可见
- 黄金链路不因基础依赖随机失败

### 工作流 D：外部数据与 ERP 最小真实联调

目标：

- 完成最小真实数据输入和执行反馈输出

交付物：

- Amazon 最小真实采集
- Google Trends 最小真实采集
- OMS / CRM / FMS / BI 最小回流
- 样本数据规范与路径约束

验收：

- 至少 2 个外部源和 4 个内部域可以提供真实或准真实输入
- 不再依赖路径随意、样本缺失的临时数据

### 工作流 E：前端与 BFF 真实交互收口

目标：

- 让工作台从“页面齐全”变成“交互可验”

交付物：

- 选品工作台真实链路联调
- Agent 平台状态联调
- 报告中心下载联调
- BFF 关键接口稳定版

验收：

- Playwright smoke 覆盖主链路
- 前端页面不再仅依赖字符串断言证明完成

### 工作流 F：质量与测试体系升级

目标：

- 让测试结果能真实反映系统可交付能力

交付物：

- 修复当前可信回归失败项
- 三层测试分组
- Playwright 联调 smoke
- real Postgres / Qdrant 专项回归

验收：

- 黄金链路回归通过
- 真实依赖相关测试具备稳定通过标准

## 13. 风险与应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 模式口径继续混乱 | “已完成”但实际仍是 mock | 以模式矩阵和状态接口统一口径 |
| 真实依赖不稳定 | Postgres / Qdrant / Kafka 随机失败 | 先做单节点基线和健康脚本 |
| 服务过重导致改动风险高 | `SelectionTaskService` 持续膨胀 | 先拆应用服务、审批服务、反馈服务 |
| 前端验收失真 | 页面存在但交互不通 | 强制加入 Playwright 联调 smoke |
| 过早推进 HA / K8s | 投入大但闭环仍未跑通 | 本阶段明确暂缓，仅保留接口和脚本 |
