## 2. 验收等级与状态定义

### 2.1 验收等级

- `A级`：阻塞项。未通过则所在阶段不得判定完成。
- `B级`：重要项。原则上必须通过；如未通过，必须有明确遗留说明和后续计划。
- `C级`：规划项或外部依赖项。记录现状即可，不阻塞本地一期收口。

### 2.2 任务状态定义

- `✅已通过`：功能、测试、证据三者齐备。
- `🔧进行中`：已完成部分实现，但未满足完整验收条件。
- `⏳待验收`：代码或配置已提交，但尚未执行正式验收。
- `🚫阻塞`：依赖外部环境、凭证、网络或联调窗口。
- `📋观察项`：不纳入当前本地一期收口，但保留后续验收入口。

### 2.3 一期通过口径

一期本地通过的最低要求：

- `P0 ~ P8` 中所有 `A级` 项全部通过
- `P9-01`、`P9-02`、`P9-03` 全部通过
- `P9-04 staging 联调` 可因外部环境阻塞暂不计入一期本地通过
- `P9-05 物理微服务拆分评估` 不计入一期范围

## 3. 通用验收原则

1. 每个任务至少同时满足“功能实现”“自动化验证或可重复手工验证”“证据留存”三项条件。
2. 对外暴露的 REST 接口任务，必须至少提供：
   - OpenAPI 文档可见
   - 成功路径示例
   - 失败路径示例
   - 权限/租户边界验证
3. 涉及数据库变更的任务，必须同时提交：
   - Alembic 迁移
   - 模型变更
   - 最少一条回归测试
4. 涉及跨域编排的任务，不只验接口返回，还要验完整链路结果。
5. 涉及兼容改造的任务，必须验证旧入口不被破坏。
6. 涉及外部系统联调的任务，在外部环境不可得时，可先验：
   - 本地 mock/local-real
   - DTO 契约
   - 联调清单
   - 阻塞说明

## 4. 通用证据要求

每项任务至少保留一种证据；A级任务至少保留两种。

| 证据类型 | 说明 |
| --- | --- |
| 代码实现 | 对应 `endpoint/service/repository/model` 文件已提交 |
| 自动化测试 | `pytest`、脚本验收、契约测试、集成测试 |
| 手工验证 | `curl` / Postman / Swagger 调用记录 |
| 运行态证据 | 服务日志、状态接口、数据库查询结果 |
| 文档证据 | README、迁移说明、OpenAPI、联调说明 |
| 工件 | `artifacts/`、导出样例、截图、JSON 结果 |

推荐证据命名方式：

- `artifacts/erp_acceptance/<task_id>/...`
- `tests/test_<domain>_api.py`
- `tests/test_erp_workflow_api.py`
- `tests/test_erp_closing_loop_integration.py`

## 7. 一期收口检查清单

一期本地收口前必须逐条确认：

- `P0` 全通过
- `P1` 全通过
- `P2` 全通过，`P2-05` 至少完成本地 DTO 验证
- `P3` 全通过，`P3-05` 至少完成本地 DTO 验证
- `P4` 全通过，`P4-05` 至少完成本地 DTO 验证
- `P5` 全通过，`P5-04` 至少完成本地回流链路验证
- `P6` 全通过，`P6-04` 至少完成口径统一验证
- `P7` 全通过
- `P8-01 ~ P8-03` 全通过
- `P9-01 ~ P9-03` 全通过

若要宣告“ERP 一期业务闭环已完成”，还必须额外满足：

- 能从 `SelectionTask` 生成 `MS product draft`
- 能基于商品草稿生成 `SCM purchase order`
- 能在 `WMS` 中完成库存预留或库容准备
- 能在 `OMS` 中生成 `Listing draft`
- 能完成订单与客户反馈回流
- 能在 `BI` 中看到闭环 KPI 与闭环总览
- 旧 `/integration` 兼容入口未失效

## 8. 推荐验收执行顺序

1. 先验 `P0`，确认骨架和接口规范稳定
2. 再验 `P1 ~ P3`，确认商品、采购、库存可运行
3. 再验 `P4 ~ P6`，确认上架、订单、反馈、BI 可运行
4. 再验 `P7`，确认闭环编排可运行
5. 最后验 `P8 ~ P9`，确认迁移兼容、测试和文档收口
