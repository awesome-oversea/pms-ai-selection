## 1. 架构概述

### 1.1 架构愿景

构建一个以**ERP数据为真相、以外部数据为视野、以知识库为记忆、以Multi-Agent为决策大脑**的跨境电商选品操作系统。系统的最终形态是替代传统“人工选品会议+邮件/Excel审批”流程的**自动化利润生成引擎**。

### 1.2 架构原则

| 原则            | 说明                 | 指导意义                             |
| :------------ | :----------------- | :------------------------------- |
| **领域驱动**      | 以业务领域为核心组织代码和团队    | 按选品、采购、财务等限界上下文划分微服务             |
| **数据驱动决策**    | 所有AI决策基于实时数据和历史知识  | 数据中台是系统的感知层，知识库是记忆层              |
| **API First** | 先设计API，再实现功能       | 所有服务提供标准RESTful API和OpenAPI文档    |
| **可观测性优先**    | 所有服务必须具备监控、日志、追踪能力 | 统一使用Prometheus+EFK+OpenTelemetry |
| **安全内建**      | 安全从设计开始，不是后期添加     | 多租户隔离、数据脱敏、AI防火墙                 |
| **云原生**       | 以Kubernetes为部署平台   | 容器化、声明式配置、GitOps                 |
| **成本可控**      | AI成本可量化、可优化        | LLM路由、模型量化、Prefix Caching        |
| **自进化**       | 系统越用越智能            | 反馈闭环、模型微调、知识库更新                  |

### 1.3 关键架构决策（ADR）

| 编号          | 决策                     | 背景                     | 理由                            | 影响               |
| :---------- | :--------------------- | :--------------------- | :---------------------------- | :--------------- |
| **ADR-001** | 选择LangGraph作为Agent编排框架 | 需要复杂的有状态工作流、条件分支、人工干预  | LangGraph原生支持状态机、检查点、人工介入     | 开发团队需学习LangGraph |
| **ADR-002** | 选择Qdrant+ES混合检索        | 既需要语义检索又需要关键词精确匹配      | Qdrant擅长向量检索，ES擅长BM25关键词检索    | 需要维护两套检索引擎       |
| **ADR-003** | 选择Iceberg作为数据湖格式       | 需要支持ACID、时间旅行、Schema演进 | Iceberg是业界标准，与Spark/Flink集成良好 | 需要学习Iceberg运维    |
| **ADR-004** | 选择Feast作为特征平台          | 需要统一离线/在线特征，保证一致性      | Feast是领先的特征存储，支持批流一体          | 需要额外维护Feast服务    |
| **ADR-005** | 选择Kong作为API网关          | 需要认证、限流、路由、灰度能力        | Kong功能全面，插件丰富，性能优异            | 需要学习Kong配置       |
| **ADR-006** | 选择Ollama+Qwen2.5作为本地模型 | 需要低成本、低延迟、数据安全的本地推理    | Ollama部署简单，Qwen2.5中文能力强       | 推理能力弱于云端大模型      |
| **ADR-007** | 选择Neo4j作为图数据库          | 需要表达供应商-商品-专利等复杂关系     | Neo4j是图数据库领导者，Cypher查询强大      | 需要学习图数据建模        |
| **ADR-008** | 选择Kafka作为统一消息总线        | 需要解耦数据采集、处理、消费         | Kafka吞吐量高、持久化、生态丰富            | 需要维护Kafka集群      |

### 1.4 架构核心公式

> **AI选品系统 = (数据中台 + RAG知识系统) × Multi-Agent决策引擎 ÷ ERP闭环执行**

- **数据中台**：感知层，实时捕获内外部数据，加工为核心特征
- **RAG知识系统**：记忆层，沉淀企业经验，越用越强
- **Multi-Agent决策引擎**：决策层，模拟专家团队协同决策
- **ERP闭环执行**：执行层，将决策转化为业务行动，形成自进化飞轮

#### 5.2.2 Agent状态机设计（LangGraph）

python

```
from langgraph.graph import StateGraph, END
from typing import TypedDict, List, Optional

class SelectionState(TypedDict):
    task_id: str
    target_market: str
    category: str
    target_roi: float
    
    # Agent输出
    collected_data: Optional[dict]
    market_analysis: Optional[dict]
    product_plan: Optional[dict]
    commercial_analysis: Optional[dict]
    risk_assessment: Optional[dict]
    
    # 控制
    current_step: str
    human_intervention_required: bool
    errors: List[str]

def build_workflow() -> StateGraph:
    workflow = StateGraph(SelectionState)
    
    # 添加节点
    workflow.add_node("data_collection", data_collection_node)
    workflow.add_node("market_insight", market_insight_node)
    workflow.add_node("product_planning", product_planning_node)
    workflow.add_node("commercial_analysis", commercial_analysis_node)
    workflow.add_node("risk_assessment", risk_assessment_node)
    workflow.add_node("report_generation", report_generation_node)
    workflow.add_node("human_review", human_review_node)
    
    # 设置入口
    workflow.set_entry_point("data_collection")
    
    # 添加边
    workflow.add_edge("data_collection", "market_insight")
    workflow.add_edge("market_insight", "product_planning")
    workflow.add_edge("product_planning", "commercial_analysis")
    workflow.add_edge("commercial_analysis", "risk_assessment")
    
    # 条件分支
    workflow.add_conditional_edges(
        "risk_assessment",
        should_require_human_review,
        {
            "skip": "report_generation",
            "review": "human_review"
        }
    )
    
    workflow.add_edge("human_review", "report_generation")
    workflow.add_edge("report_generation", END)
    
    return workflow.compile()
```

### 10.2 技术选型理由

| 技术                         | 对比方案            | 选型理由                          |
| :------------------------- | :-------------- | :---------------------------- |
| **LangGraph vs LangChain** | LangChain Agent | LangGraph状态机更适合有状态、多分支的选品流程   |
| **Qdrant vs Milvus**       | Milvus          | Qdrant Rust实现性能更好、过滤能力更强      |
| **Iceberg vs Delta Lake**  | Delta Lake      | Iceberg生态更开放、与Flink/Spark集成更好 |
| **Feast vs 自研**            | 自研特征服务          | Feast成熟度高、避免重复造轮子             |
| **Kong vs Nginx**          | Nginx Ingress   | Kong插件生态丰富、API管理功能全面          |
| **Ollama vs 自建**           | 自建推理服务          | Ollama部署简单、GGUF量化、成本低         |
| **Neo4j vs PostgreSQL**    | PostgreSQL图查询   | Neo4j图遍历性能优、Cypher表达力强        |

### 10.3 技术风险评估

| 技术            | 风险                  | 影响 | 缓解措施                |
| :------------ | :------------------ | :- | :------------------ |
| **LangGraph** | 框架较新，社区生态待完善        | 中  | 关注版本更新，准备降级方案       |
| **vLLM**      | Prefix Caching内存占用高 | 中  | 监控内存使用，配置swap空间     |
| **Iceberg**   | 小文件问题               | 中  | 定期compaction，配置合并策略 |
| **Feast**     | 学习曲线陡峭              | 低  | 提前培训，编写最佳实践文档       |
| **Kafka**     | 运维复杂                | 中  | 使用托管服务或加强运维培训       |
| **Neo4j**     | 大规模数据性能             | 中  | 合理建模，使用索引，读写分离      |
| **Ollama**    | 推理能力弱于云端            | 低  | 复杂任务路由到vLLM或云端API   |
