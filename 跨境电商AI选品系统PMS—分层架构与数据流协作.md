## 跨境电商AI选品系统PMS——分层架构与数据流协作

> 本部分详细阐述跨境电商AI选品系统中**数据流向**、**数据获取模式**、**爬虫技术实现**、**Agent与数据的交互细节**、**内部系统闭环关系**，以及**架构层之间的区别与联系**。以“户外储能电源选品”为业务背景，确保概念清晰、可直接指导实现。

***

### 一、内部业务系统职责与数据内容

| 系统      | 英文全称                             | 核心职责       | 关键数据内容                                   | 与AI选品系统的交互                                         |
| :------ | :------------------------------- | :--------- | :--------------------------------------- | :------------------------------------------------- |
| **OMS** | Order Management System          | 订单管理、销售履约  | 订单明细（SKU、数量、金额、时间）、订单状态、退款记录、促销活动数据      | 提供历史销量用于市场趋势分析；接收AI采纳后自动创建的Listing草稿；通过CDC推送订单数据回流 |
| **WMS** | Warehouse Management System      | 仓储管理、库存调度  | 实时库存量、库龄、周转率、入库/出库记录、库容利用率               | 提供库存水平避免超卖；接收AI采纳后预留库容；反馈库存滞销/缺货风险                 |
| **SCM** | Supply Chain Management          | 供应链管理、采购协同 | 供应商信息（交期、质量评分、价格）、采购订单、采购历史、物流跟踪         | 提供供应商可靠性数据；接收AI采纳后自动创建采购单；反馈采购到货状态                 |
| **CRM** | Customer Relationship Management | 客户关系、评价与客诉 | 客户评价（评分、评论文本）、客诉记录、退换货原因、客户画像            | 提供用户反馈用于产品痛点挖掘和风险评估；通过CDC推送评价数据回流，更新知识库            |
| **FMS** | Financial Management System      | 财务管理、成本核算  | 头程运费、关税、FBA费用、广告费、其他杂费；产品成本、毛利率、净利       | 提供全链路成本数据，支撑商业化Agent进行利润测算和定价建议                    |
| **BI**  | Business Intelligence            | 商业智能、报表分析  | 历史KPI（爆款命中率、ROI、选品周期）、广告转化率（ACOS）、销售趋势报表 | 提供历史选品绩效数据，用于报告生成中的“预测vs实际”对比；支持管理者看板              |

### 三、两种数据获取模式

| 模式            | 适用场景                   | 技术实现                          | 与Agent的关系                      |
| :------------ | :--------------------- | :---------------------------- | :----------------------------- |
| **离线/近线批量导入** | 历史数据、每日聚合特征、不要求秒级实时    | Kafka → Flink/Spark → 数据湖/特征库 | Agent在执行时查询已处理好的特征（如日销量、7天增长率） |
| **实时在线拉取**    | 竞品当前价格、最新评论、TikTok热门视频 | Agent直接调用外部API或内部系统接口         | Agent主动发起请求，获取最新数据用于当前分析       |

### 四、爬虫技术实现

#### 4.1 爬虫引入的价值与场景

| 爬虫目标                             | 数据内容                         | 业务价值                        |
| :------------------------------- | :--------------------------- | :-------------------------- |
| **竞品官网**                         | 产品详情、价格、促销信息、用户评价（非Amazon平台） | 获取更全面的竞品动态，弥补官方API覆盖不足      |
| **跨境电商论坛（Reddit、SellerCentral）** | 卖家讨论、痛点吐槽、热销品推荐              | 发现早期趋势和潜在蓝海品类               |
| **社交媒体（Instagram、Pinterest）**    | 商品图片、标签、互动数据                 | 捕捉视觉流行趋势，辅助产品规划Agent的多模态分析  |
| **比价网站（CamelCamelCamel）**        | 历史价格曲线、排名变化                  | 验证Amazon API数据的准确性，获取更长历史窗口 |
| **行业媒体/博客**                      | 深度评测、新品发布、技术趋势               | 丰富RAG知识库，提升风险评估和市场洞察        |
| **专利/商标局公开页面**                   | 专利全文、法律状态                    | 补充专利数据库，增强侵权检查覆盖度           |

**关键组件说明**：

- **任务调度器**：基于Airflow/Prefect管理爬虫触发频率（每日/每小时），支持增量采集和断点续爬。
- **去重过滤器**：使用BloomFilter对URL和内容哈希进行去重，避免重复采集。
- **数据清洗**：解析HTML/JSON，提取结构化字段，进行Schema校验和异常值过滤。
- **爬虫节点**：Scrapy处理静态页面，Playwright/Selenium处理动态渲染和模拟登录。
- **代理IP池**：集成住宅代理、数据中心代理，实现请求IP轮换和反反爬。
- **输出**：清洗后的数据写入Kafka或数据湖，与API采集数据统一存储。

#### 4.3 反反爬策略

| 策略               | 实现方式                                          |
| :--------------- | :-------------------------------------------- |
| **User-Agent轮换** | 维护常用UA库，每次请求随机选择                              |
| **请求头伪装**        | 添加Accept-Language、Referer等真实浏览器头              |
| **IP代理池**        | 使用付费代理服务（如BrightData、oxylabs）或自建代理池           |
| **请求延迟与随机化**     | 设置随机间隔（1\~5秒），模仿人类行为                          |
| **验证码识别**        | 对接OCR服务（Tesseract、打码平台）处理简单验证码                |
| **浏览器指纹模拟**      | Playwright/Selenium模拟真实浏览器环境，包括Canvas、WebGL指纹 |
| **登录态维持**        | 对于需要登录的网站，使用Cookie池定期刷新                       |

#### 4.4 数据质量保障

- **Schema校验**：每个爬取的数据必须符合预定义的Schema（字段类型、必填性），不符合的进入死信队列。
- **去重**：基于URL+页面内容哈希进行去重，避免重复入库。
- **异常监控**：监控爬虫任务的数据量、成功率、解析错误率，低于阈值时触发告警。
- **交叉验证**：爬取的数据与官方API数据进行对比，若偏差过大则标记为低置信度。

#### 4.5 合规与法律注意事项

- 遵守目标网站的`robots.txt`协议。
- 设置合理的请求频率（如1请求/秒），避免对源站造成压力。
- 不采集用户个人隐私数据（如姓名、地址、电话）。
- 对于商业用途，建议咨询法务，避免违反《反不正当竞争法》或网站服务条款。
- 优先使用官方API，爬虫仅作为补充手段。

| <br />        | RAG知识库中的报告模板（PDF/Excel/PPT格式规范）          | 向量检索匹配最佳模板                    | 生成符合企业品牌风格、包含图表和文字建议的专业报告         |

| <br />        | 企业微信/钉钉机器人回调地址                           | 配置读取                          | 报告生成后自动推送到运营群，实现一键分享              |
| <br />        | **OMS/WMS/SCM**：执行状态反馈                   | 通过API回调或消息队列                  | 在报告中展示执行进度，形成闭环告知                 |

### 六、为什么需要Agent主动拉取，而不全用离线数据？

| 原因        | 说明                                       |
| :-------- | :--------------------------------------- |
| **时效性要求** | 竞品价格变动、TikTok爆款视频出现后，需要在分钟级内响应，离线批处理无法满足 |
| **个性化查询** | 每个选品任务关注的点不同（如特定ASIN、特定关键词），无法预先计算所有特征   |
| **成本控制**  | 不是所有数据都需要永久存储（如某次TikTok视频分析结果），按需拉取更经济   |
| **爬虫补充**  | 无API的网站数据必须通过爬虫获取，且爬虫更适合批量离线采集，而非实时在线拉取  |

### 七、内部系统与Agent的闭环关系（完整版）

text

```
Agent决策（推荐商品） → 运营“一键采纳”
         │
         ▼
API调用SCM创建采购单（含供应商、数量、期望交期）

         │
         ▼
API调用OMS创建Listing草稿（标题、描述、价格、图片）
         │
         ▼
OMS反馈Listing草稿ID
         │
         ▼
商品上架销售
         │
         ▼
OMS产生订单数据（销量、销售额、退款）
CRM产生评价数据（评分、评论文本、客诉）
         │
         ▼
CDC实时捕获 → 写入Kafka → Flink消费
         │
         ├─→ 更新特征库（销量增长率、评价情感得分）
         ├─→ 更新向量库（新评价的向量表示）
         └─→ 更新RAG知识库（将好评/差评作为案例入库）
         │
         ▼

         │
         ▼
下次选品时，Agent可检索到：
- 过往相似商品的真实表现（来自OMS/BI）
- 用户对类似功能的评价（来自CRM）
- 供应商实际交付表现（来自SCM）
- 库存周转影响（来自WMS）
- 爬虫补充的竞品动态（来自数据湖）
         │
         ▼

```

### 八、详解架构层各层的区别与联系

#### 8.1 分层职责矩阵（含内部系统映射）

| 层            | 核心职责              | 输入            | 输出               | 依赖的下层           | 关联的内部系统                       |
| :----------- | :---------------- | :------------ | :--------------- | :-------------- | :---------------------------- |
| **用户接入层**    | 多端适配（浏览器/App/机器人） | 用户操作          | HTTP/WebSocket请求 | 网关层（认证）         | -                             |
| **网关层**      | 安全、流量管控           | 原始请求          | 过滤后的请求           | API层（转发）        | -                             |
| **UI层**      | 可视化交互             | 用户输入          | 展示数据             | API层（通过HTTP/WS） | 调用OMS/WMS/SCM的展示接口            |
| **API层**     | 接口契约、任务管理         | 外部请求          | 业务响应             | Agent编排层、数据存储层  | 封装对OMS/WMS/SCM/CRM/FMS/BI的调用  |
| **Agent编排层** | 智能决策流程            | 任务参数          | 决策结果             | AI中台、数据处理层      | 通过API层间接调用内部系统                |
| **AI中台**     | AI能力服务化           | Agent调用       | 模型推理/RAG结果       | LLM模型层、推理引擎层    | -                             |
| **LLM模型层**   | 模型管理与路由           | 推理请求          | 模型输出             | 推理引擎层           | -                             |
| **推理引擎层**    | 模型高性能执行           | 模型+输入         | 张量结果             | 基础设施（GPU）       | -                             |
| **数据处理层**    | 数据清洗、聚合、特征        | 原始数据（Kafka/湖） | 特征/向量            | 数据存储层           | 消费OMS/WMS/SCM/CRM/FMS的CDC数据   |
| **数据存储层**    | 持久化与缓存            | 读写请求          | 数据               | 基础设施（存储）        | 存储OMS/WMS/SCM/CRM/FMS/BI的数据副本 |
| **外部/内部集成**  | 适配异构系统            | 外部API/内部接口    | 标准化数据            | 无（横向依赖）         | 包含对OMS/WMS/SCM/CRM/FMS/BI的适配器 |
| **监控运维**     | 观测与控制             | 各层指标/日志       | 告警/仪表盘           | 所有层             | 监控内部系统的API健康度                 |
| **基础设施**     | 计算/网络/存储资源        | 容器/虚拟机        | 运行环境             | 无               | -                             |

#### 8.2 层与层之间的关键联系

**垂直调用链（同步请求路径）**

text

```
用户 → 网关层（Kong）→ API层（FastAPI）→ Agent编排层（LangGraph）→ AI中台 → LLM模型层 → 推理引擎层 → 返回结果
```

- 每一层只依赖紧邻的下层，符合分层架构原则。
- 内部系统（如SCM）被API层调用，不在垂直链中。

**横向依赖（数据流）**

- 数据处理层与数据存储层紧密耦合：Flink消费Kafka后写入数据湖/向量库。
- 外部/内部集成与数据处理层：外部数据通过采集（API+爬虫）写入Kafka，内部系统通过CDC写入Kafka。
- Agent编排层既依赖AI中台（同步推理），也依赖数据处理层产出的特征（异步查询）。

**层次隔离的好处**

- **更换推理引擎**：从vLLM换成TGI，只需修改AI中台中的推理引擎层配置，上层Agent无感知。
- **增加新数据源**：只需在外部集成层增加适配器（API或爬虫），写入Kafka，下游Flink自动处理，Agent无需修改。
- **升级模型**：在LLM模型层增加新模型版本，AI中台的路由策略可灰度切换。
- **替换内部系统**：若更换OMS供应商，只需修改集成层的适配器，上层Agent和API层无感知。

#### 8.3 容易混淆的两对层次

| 混淆对                  | 区别                                                              | 联系                 |
| :------------------- | :-------------------------------------------------------------- | :----------------- |
| **AI中台 vs Agent编排层** | AI中台提供原子能力（推理、检索、向量化）；Agent编排层负责流程组合（先查数据、再调用LLM、最后生成报告）        | Agent编排层调用AI中台的API |
| **数据处理层 vs 数据存储层**   | 数据处理层是**计算**（Flink/Spark）；数据存储层是**存储**（PostgreSQL/Qdrant/Kafka） | 数据处理层读写数据存储层       |
