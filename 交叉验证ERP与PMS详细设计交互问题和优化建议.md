## 二、总体交叉验证结论

ERP 与 PMS 的总体架构方向正确，但仍需持续收敛边界与落地口径。

### 2.1 正确定位

| 系统 | 正确定位 |
|---|---|
| ERP | 经营主系统、正式业务数据真相源、审批与执行层、权限和审计主控 |
| PMS | 外部数据采集、AI选品分析、跨域经营建议、智能决策辅助与证据链管理系统 |

### 2.2 推荐总体交互模式

```text
外部市场数据 + ERP经营数据
        ↓
PMS数据采集 / 特征加工 / RAG / Multi-Agent分析
        ↓
PMS生成选品、定价、广告、补货、风险、产品优化建议
        ↓
ERP建议池 / 草稿单据 / 待审批动作
        ↓
ERP权限校验 / 业务规则校验 / 审批流
        ↓
ERP各业务域执行正式业务动作
        ↓
执行结果 / KPI / 销售 / 库存 / 利润 / 广告效果反馈
        ↓
PMS复盘、模型优化、知识库更新
```

### 2.3 当前主要问题

1. ERP 与 PMS 的主从关系已在 V11 中强化，但历史版本中仍存在 PMS 直接写 ERP 终态数据的描述，需要清理。
2. ERP V3/V6/V7/V8/V11 与 PMS V6/V8/V11 多版本并存，部分章节仍混有旧口径。
3. ERP 项目开发方案强调一期“最小经营闭环”，但 ERP V11/PMS V11 已包含 14 域、AI、CDC、事件、微服务等目标态能力，存在范围膨胀风险。
4. PDM 与 PMS 在选品、竞品分析、产品机会、产品立项上边界仍需冻结。
5. AI 能力在 ERP 多域与 PMS 中重复出现，需要明确“PMS跨域AI决策中心、ERP领域执行和规则中心”。
6. IAM 数据权限维度在 V11 中已增强，但需要落实到所有 PMS 调用、Agent任务、后台任务、知识库/向量库访问。
7. API、CDC、事件、BI 指标接口的使用场景需要矩阵化，避免所有数据都 API 或所有数据都 CDC。
8. 建议执行状态机已有设计，但 ERP/PMS 双向状态一致性、幂等、补偿、失败重试仍需强化。

## 三、ERP 详细设计问题及优化建议

### 3.1 ERP 版本口径仍需明确冻结

#### 问题

ERP 文档存在多个版本：V3/V6/V7/V8/V11。V11 已声明为当前主口径，但历史文档仍可能被开发、测试或产品人员引用。

| 版本 | 主要特征 | 当前建议定位 |
|---|---|---|
| ERP V3/V6 | 基础ERP能力，PMS/AI集成较弱 | 历史参考 |
| ERP V7 | 开始引入 PMS/AI选品 | 演进参考 |
| ERP V8 | 14域、PMS集成、CDC、API、AI能力较完整 | V11基础来源 |
| ERP V11 | 基于交叉验证优化，合并 V9/V10 口径 | 当前详细设计主口径 |
| ERP需求规格说明书 | 业务需求基线 | 需求约束 |
| ERP项目开发方案 | 阶段范围与实施边界 | 项目实施约束 |

#### 优化建议

- 在 ERP V11 首页明确：**ERP V11 是当前唯一详细设计主口径**。
- ERP V3/V6/V7/V8 只保留为历史演进来源，不作为开发实现依据。
- 开发任务、接口设计、测试用例、验收标准均引用 ERP V11。
- 对历史版本中冲突描述建立“废弃口径清单”。

---

### 3.2 ERP 项目范围与详细设计目标态存在膨胀风险

#### 问题

`跨境ERP-项目开发方案.md` 明确一期聚焦最小经营闭环：店铺接入、商品、SKU映射、订单、库存、采购、履约、售后、基础结算与利润分析。

但 ERP V11 中包含：

- 14 个业务域；
- AI能力分布；
- CDC数据同步；
- PMS 14域集成；
- 多租户、微服务、事件驱动、开放平台等目标态能力。

如果没有实现阶段标注，会导致一期范围失控。

#### 优化建议

| 阶段 | ERP应交付重点 | PMS集成重点 |
|---|---|---|
| 一期 | IAM、PDM、SOM、OMS、WMS、SCM、FMS基础闭环 | PMS建议池、选品建议、基础经营数据读取 |
| 二期 | ADS、FBA、TMS、CRM、BI增强 | 广告优化、补货预测、客户反馈、KPI回流 |
| 三期 | CDC、事件驱动、多租户、AI服务化 | 建议闭环、模型复盘、跨域Agent编排 |
| 四期 | 开放平台、插件生态、智能服务市场 | PMS作为外部智能服务生态节点 |

每个 ERP V11 模块建议补充字段：

```text
实现阶段：一期 / 二期 / 三期 / 四期
实现状态：已实现 / 本期计划 / 目标态 / 外部依赖
是否PMS依赖：是 / 否
```

---

### 3.3 ERP 作为主系统的表达还需贯穿全域

#### 问题

ERP V11 已明确“ERP保存正式业务主数据、业务单据、库存、订单、财务和执行结果”，但部分章节仍可能使用“AI驱动执行层”“PMS自动执行”等表达。

风险：

- PMS 越权创建正式采购单、Listing、广告活动、补货单；
- ERP 审批流被绕过；
- 审计责任不清；
- 财务、库存、订单结果难以追责。

#### 优化建议

统一 ERP 全文口径：

```text
PMS只能创建建议、草稿、待审批动作、风险预警、洞察卡片。
ERP负责正式业务对象创建、审批、执行、作废、回滚和审计。
```

ERP 侧建议增加统一接收对象：

| 对象 | 用途 |
|---|---|
| Recommendation | AI建议，如选品、定价、广告、补货 |
| Draft | 草稿单据，如Listing草稿、采购草稿 |
| PendingAction | 待审批动作，如调价、投放调整 |
| RiskAlert | 风险预警，如侵权、断货、利润异常 |
| InsightCard | 经营洞察卡片，供 BI/工作台展示 |

### 3.4 ERP IAM 权限维度需要落到接口与数据层

#### 问题

ERP V11 与 PMS V11 都强化了 IAM 权限维度，但设计中仍需明确这些维度如何进入：

- API 请求；
- 数据查询过滤；
- Agent 执行上下文；
- 知识库/向量库检索；
- 审计日志；
- BI 指标聚合。

#### 优化建议

ERP IAM 统一权限维度：

| 权限维度 | 示例 |
|---|---|
| tenant | 租户 |
| org | 公司/组织 |
| department | 部门 |
| store | 店铺 |
| marketplace | Amazon US、Amazon JP |
| channel | Amazon、TikTok、Walmart |
| warehouse | 国内仓、FBA仓、海外仓 |
| supplier | 供应商 |
| category | 类目 |
| data_level | 明细、汇总、脱敏 |

所有 PMS 调用 ERP 的接口必须包含：

```text
tenant_id
actor_id
actor_type = user / service_account / agent
scope
purpose
trace_id
idempotency_key
source_system = PMS
```

### 3.5 ERP API 路径与接口分层仍需严格统一

#### 问题

历史文档中存在 `/api/v1/...`、`/api/admin/v1/...`、`/api/open/v1/...`、`/api/internal/v1/...` 等多种路径。V11 已倾向统一，但需要明确 PMS 调用边界。

#### 优化建议

| API路径 | 用途 | PMS是否使用 |
|---|---|---|
| `/api/admin/v1` | ERP管理后台 | 否 |
| `/api/open/v1` | 外部开放接口 | 可选，需授权 |
| `/api/internal/v1` | 内部系统接口 | PMS优先使用 |
| `/api/pms/v1` | PMS专用适配接口 | 可作为兼容层 |
| `/api/v1` | 普通业务前端接口 | PMS不应直接使用 |

PMS 写入类接口必须具备：

- 幂等键；
- 业务校验；
- 权限校验；
- 审计日志；
- 状态回调；
- 失败补偿；
- 重放保护。

---

### 3.6 ERP 订单链路要保持完整，避免简化影响AI判断

#### 问题

ERP 需求规格强调订单链路完整，项目开发方案也要求“渠道订单、ERP销售单、履约单三层链路完整”。PMS 做销量预测、退货风险、履约风险、利润分析时必须依赖完整订单链路。

#### 优化建议

OMS 需保持以下对象：

```text
平台订单 ChannelOrder
ERP销售订单 SalesOrder
履约单 FulfillmentOrder
包裹 Package
物流运单 Shipment
售后/退款/退货/重发 AfterSales
订单异常 OrderException
风控结果 RiskResult
```

PMS 不直接修改订单状态，只读取订单、退货、异常、履约表现，并写入风险预警或策略建议。

## 四、PMS 详细设计问题及优化建议

### 4.1 PMS V11 已强化边界，但历史文档中的侵入式执行描述需清理

#### 问题

PMS V11 明确“PMS不写ERP终态”，但 PMS V6/V8、架构方案、业务设计中可能仍存在：

- AI采纳后自动创建 Listing；
- 自动创建采购单；
- 自动调整广告预算；
- 自动发起补货计划；
- 自动修改价格。

这些描述如果未清理，会与 V11 主口径冲突。

#### 优化建议

将所有“自动创建正式单据/正式业务动作”改为：

| 原描述 | 修正描述 |
|---|---|
| 自动创建Listing | 创建Listing草稿或Listing优化建议，由ERP SOM审批执行 |
| 自动创建采购单 | 创建采购建议或采购草稿，由ERP SCM审批执行 |
| 自动调整广告预算 | 创建广告优化建议，由ERP ADS审批或规则自动执行 |
| 自动发起补货计划 | 创建补货建议，由ERP SCM/WMS/FBA校验执行 |
| 自动修改价格 | 创建定价建议，由ERP SOM审批执行 |

---

### 4.2 PMS 目标态、近期态、当前态仍需逐模块标注

#### 问题

PMS V11 合并了 V9/V10，覆盖范围完整，但模块很多：ERP 14域集成、Agent、RAG、LLM、数据湖、CDC、Kafka、Flink、知识图谱、报告中心、前端工作台等。

若不逐模块标注状态，开发任务会失焦。

#### 优化建议

每个 PMS 服务、接口、表、Agent、事件都标注：

| 状态 | 含义 |
|---|---|
| 已实现 | 当前代码或当前版本已具备 |
| 本期实现 | 当前迭代必须交付 |
| 近期实现 | 下一阶段交付 |
| 目标态 | 架构规划，暂不实现 |
| 外部依赖 | 依赖 ERP 或第三方系统能力 |

---

### 4.3 PMS 数据源可信等级需要强制进入评分模型

#### 问题

PMS 数据来源多，包括外部平台、第三方工具、爬虫、社媒、新闻、专利、ERP经营数据、LLM推断。不同来源可信度差异大。

如果推荐评分未体现数据可信等级，容易出现：

- 用低可信爬虫数据覆盖 ERP 真实数据；
- 用社媒热度过度推断销量；
- LLM幻觉被当作事实；
- 选品建议缺乏可解释性。

#### 优化建议

| 数据源 | 可信等级 | 用途 |
|---|---|---|
| ERP订单/库存/财务/采购 | A | 经营事实、利润、库存、补货依据 |
| 官方平台API | A/B | 销量、广告、Listing、库存依据 |
| 第三方数据服务 | B | 市场容量、竞品趋势参考 |
| 爬虫数据 | C | 补充采样，需要去重和校验 |
| 社媒趋势 | C | 趋势信号，不可单独决策 |
| LLM推断 | D | 解释、总结、假设，不作为事实源 |

PMS 推荐输出必须包含：

```text
score
evidence
data_sources
confidence
risk_flags
explainability
assumptions
missing_data
```

---

### 4.4 PMS 权限模型需覆盖 Agent、后台任务、知识库与向量库

#### 问题

PMS V11 已提出双主体模型，但需要细化：

- 用户触发的 Agent 是否继承用户权限；
- 定时任务使用哪个服务账号；
- 知识库/向量库按租户还是按业务域隔离；
- 报告是否脱敏；
- Agent 是否可跨店铺、跨市场读取数据。

#### 优化建议

统一执行主体模型：

```text
用户主体：user_id + roles + scopes + data_permissions
服务主体：service_account_id + allowed_domains + allowed_actions
Agent主体：agent_id + delegated_actor + execution_scope + purpose
```

所有 Agent 执行上下文必须包含：

```text
tenant_id
actor_type
actor_id
agent_id
scope
purpose
data_level
trace_id
```

知识库、向量库、特征库、报告中心均按以下维度过滤：

```text
tenant_id + store + marketplace + channel + category + data_level
```

---

### 4.5 PMS 对 ERP 14域集成需区分 API、事件、CDC、BI 指标

#### 问题

PMS 文档中存在 API、CDC、Kafka、BI、数据湖多种集成方式。如果没有规则，开发会混用。

#### 优化建议

| 数据类型 | 推荐接入方式 | 示例 |
|---|---|---|
| 实时决策数据 | ERP Internal API | 可售库存、当前价格、审批状态 |
| 高频变更数据 | CDC/领域事件 | 库存变动、订单状态、广告消耗 |
| 汇总分析数据 | BI/指标API | GMV、毛利率、ACOS、库存周转 |
| 大批量历史数据 | 数据湖/离线同步 | 历史订单、历史广告、历史库存 |
| 建议执行反馈 | 事件 + API查询 | 建议采纳、拒绝、执行结果 |

## 五、两个系统交互设计问题及优化建议

### 5.1 PDM 与 PMS 职责重叠

#### 问题

ERP PDM 与 PMS 都涉及选品、竞品、产品机会、产品分析、产品开发。若不冻结边界，会导致两个系统都建设选品工作台和产品机会池。

#### 优化建议

| 能力 | 主责系统 | 说明 |
|---|---|---|
| 外部市场采集 | PMS | Amazon、TikTok、1688、Trends、社媒、竞品 |
| AI选品评分 | PMS | 生成评分、证据链、风险、假设 |
| 选品建议 | PMS | 输出 Recommendation |
| 产品机会池 | PMS主导，ERP可展示 | PMS保存机会分析，ERP展示采纳入口 |
| 产品立项 | ERP PDM | 正式立项、编号、负责人、流程 |
| SPU/SKU主数据 | ERP PDM | ERP是真相源 |
| 质检/IP/合规 | ERP PDM | ERP承接流程与记录 |
| 产品开发流程 | ERP PDM | 审批、任务、状态、结果 |

结论：

```text
PMS负责“发现与建议”，ERP PDM负责“立项与开发”。
```

---

### 5.2 Listing 归属必须统一为 SOM

#### 问题

早期 PMS/ERP 设计中存在 OMS 创建 Listing 草稿的描述。OMS 是订单域，不应处理 Listing。

#### 优化建议

```text
PMS选品建议被采纳
    ↓
ERP PDM创建产品开发任务
    ↓
ERP SOM创建Listing草稿
    ↓
ERP ADS创建广告建议/广告草稿
    ↓
ERP SCM/WMS/FBA创建采购与补货建议
```

OMS 仅负责订单、履约、退款、异常、风控，不负责 Listing 创建。

---

### 5.3 ADS 与 SOM 边界需要继续强化

#### 问题

历史文档中广告能力有时放在 SOM，有时独立为 ADS。PMS 广告优化建议可能不知道写入 ADS 还是 SOM。

#### 优化建议

| 域 | 职责 |
|---|---|
| SOM | Listing、价格、渠道SKU、销售策略 |
| ADS | 广告活动、关键词、预算、竞价、ACOS、ROAS |
| FMS | 广告费用归集、利润核算 |
| BI | 广告KPI分析与趋势报表 |
| PMS | 广告优化建议、预算建议、关键词建议 |

PMS 广告建议进入 ADS；广告成本与利润效果由 FMS/BI 反馈。

### 5.4 AI能力分布需要避免重复建设

#### 问题

ERP V11 中多个域标注 AI：PDM AI选品、ADS AI优化、OMS AI风控、SCM AI补货、WMS AI预测、CRM AI情感、BI KPI。PMS V11 也包含对应 Agent 能力。

风险：

- 重复建设模型；
- 推荐冲突；
- 特征口径不一致；
- 训练数据重复；
- 责任边界不清。

#### 优化建议

| AI能力 | PMS职责 | ERP职责 |
|---|---|---|
| AI选品 | 跨域选品评分、证据链 | PDM立项承接 |
| AI广告优化 | 预算/关键词/竞价建议 | ADS审批与执行 |
| AI风控 | 风险信号识别、解释 | OMS规则校验与处置 |
| AI补货 | 补货预测与建议 | SCM/WMS/FBA执行 |
| AI情感 | 评论情绪与产品痛点 | CRM售后处理 |
| AI KPI | 经营洞察生成 | BI指标口径与评估 |

统一原则：

```text
PMS = 跨域AI决策中心
ERP = 领域规则 + 审批 + 执行 + 反馈
```

### 5.5 数据主权需以 ERP 为经营事实源

#### 问题

PMS 会保存选品任务、AI推荐、评分、证据链、特征库、向量库、知识库、同步日志。ERP 保存正式业务数据。若不明确主权，会出现 SKU、Listing、成本、利润、KPI 口径冲突。

#### 优化建议

| 数据 | 主系统 | PMS可保存内容 |
|---|---|---|
| 商品主数据 | ERP PDM | 商品快照、分析特征 |
| SPU/SKU | ERP PDM | SKU映射快照、评分特征 |
| Listing | ERP SOM | Listing表现快照、优化建议 |
| 广告活动 | ERP ADS | 广告表现特征、优化建议 |
| 订单 | ERP OMS | 订单汇总特征、风险分析 |
| 库存 | ERP WMS/FBA | 库存快照、预测特征 |
| 采购 | ERP SCM | 供应商评分、采购建议 |
| 物流 | ERP TMS | 运费/时效特征、风险建议 |
| 售后 | ERP CRM | 情绪分析、痛点聚合 |
| 成本利润 | ERP FMS | 利润风险评分、模拟结果 |
| KPI | ERP BI | AI洞察、复盘结论 |
| 选品任务 | PMS | 主数据 |
| AI推荐/评分/证据链 | PMS | 主数据 |
| 建议执行状态 | ERP + PMS | 双向同步 |

## 七、接口、事件、权限与状态机优化建议

### 7.1 PMS读取ERP数据

| 场景 | 推荐方式 |
|---|---|
| 查询当前库存/价格/审批状态 | `/api/internal/v1` |
| 同步订单/库存/广告高频变化 | CDC / 领域事件 |
| 获取经营指标/KPI | BI 指标 API |
| 获取历史训练数据 | 数据湖 / 批量导出 |

### 7.2 PMS写入ERP数据

PMS 只能写入：

- Recommendation；
- Draft；
- PendingAction；
- RiskAlert；
- InsightCard。

不得直接写入：

- 正式采购单；
- 正式订单；
- 正式库存流水；
- 正式财务凭证；
- 已生效广告活动；
- 已生效价格；
- 已发布 Listing。

### 7.3 建议执行状态机

```text
created
scored
submitted
approved
rejected
executing
partially_executed
executed
failed
rolled_back
measured
```

状态同步要求：

- ERP 是审批与执行状态主控；
- PMS 是建议内容、评分、证据链主控；
- 状态变更需事件通知 PMS；
- PMS 可通过 API 查询最终状态；
- 所有状态变更必须带 trace_id 与 audit_context。

### 7.4 事件回流清单

ERP 应向 PMS 回流：

- 建议提交成功；
- 建议审批通过；
- 建议审批拒绝及原因；
- 草稿转正式单据；
- 执行失败及原因；
- 执行回滚；
- 销售表现变化；
- 库存表现变化；
- 广告表现变化；
- 利润表现变化；
- KPI 复盘结果。

## 八、优先级优化建议清单

### 8.1 P0：必须冻结的设计口径

1. ERP V11 是当前主详细设计口径。
2. PMS V11 是当前主详细设计口径。
3. ERP 是经营数据真相源、审批层、执行层和审计主控。
4. PMS 是 AI 决策建议系统，不直接写 ERP 终态数据。
5. Listing 属于 SOM，不属于 OMS。
6. 产品立项与 SPU/SKU 主数据属于 ERP PDM。
7. 广告投放与预算执行属于 ERP ADS。
8. 成本利润口径属于 ERP FMS。
9. KPI 口径与建议效果评估属于 ERP BI。
10. PMS 推荐必须具备证据链、置信度、数据来源和风险标识。

### 8.2 P1：需要补齐的集成设计

1. PMS → ERP API接口矩阵。
2. ERP → PMS 事件矩阵。
3. PMS/ERP 数据主权矩阵。
4. IAM 权限矩阵。
5. 建议池/草稿/待审批动作统一模型。
6. 幂等、重试、补偿、回放设计。
7. Agent 执行上下文与审计模型。
8. BI 建议效果评估指标。

### 8.3 P2：需要落实的工程治理

1. 所有 PMS 调用 ERP 接口统一走内部接口或 PMS 专用接口。
2. 禁止 PMS 复用 ERP 前端业务 API。
3. 所有写接口必须带 idempotency_key。
4. 所有数据读取必须经过 IAM 数据权限过滤。
5. 向量库、知识库、特征库必须按 tenant/scope/data_level 隔离。
6. AI 自动执行必须支持开关、灰度、人工审批和回滚。
7. 所有 LLM 推断必须标注为推断，不可作为事实源。

## 十、最终结论

ERP 与 PMS 的整体集成方向正确，V11 已经明显吸收了前序交叉验证建议，但仍需要在实施层面继续收敛以下问题：

1. **以 ERP V11 和 PMS V11 为唯一当前主设计口径**，历史版本只作为参考。
2. **ERP 是主系统，PMS 是 AI建议系统**，PMS 不得绕过 ERP 审批与审计。
3. **PMS 只能写建议、草稿、待审批动作、风险预警和洞察卡片**，不得直接创建正式业务终态。
4. **PDM/SOM/ADS/OMS/SCM/WMS/FBA/TMS/CRM/FMS/BI/SYS 的职责边界必须冻结**。
5. **IAM 权限维度必须贯彻到 API、Agent、知识库、向量库、特征库和报告中心**。
6. **接口、事件、CDC、BI 指标必须矩阵化管理**，避免集成方式混乱。
7. **AI能力由 PMS 统一做跨域决策，ERP 各域负责规则、审批、执行和反馈**。
8. **建议执行状态机必须双向同步**，ERP 控制执行状态，PMS 控制建议内容和证据链。
9. **所有推荐结果必须可解释、可审计、可回放、可复盘**。
10. **项目实施必须区分一期闭环、二期增强、三期事件驱动/AI闭环、四期开放平台目标态**，避免范围膨胀。
