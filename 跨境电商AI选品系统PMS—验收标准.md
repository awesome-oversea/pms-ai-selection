## 2. 验收等级与状态定义

### 2.1 验收等级

- `A级`：阻塞项。未通过则对应任务不得标记为完成。
- `B级`：重要项。原则上应通过，若未通过需给出明确遗留说明和整改计划。
- `C级`：观察项。记录结果即可，不阻塞当前任务收口。

### 2.2 任务状态

- `✅已完成`：功能、证据、回归已满足当前口径
- `🔧进行中`：已部分实现，但未达到完整验收条件
- `⏳待开始`：尚未进入实质开发或联调
- `📋规划中`：已识别方向，但尚未进入排期执行

### 2.3 推进属性对应的验收策略

- `🔴本地可立即推进`：必须提供本地运行证据
- `🟡本地先做外部验证`：至少完成本地实现、自验收和外部联调准备
- `🔵外部环境阻塞`：当前可先验部署方案、配置、工件和联调预案，待环境可用后补真实验收

## 3. 通用验收原则

1. 每个任务至少要同时满足“功能完成”“证据完整”“可复核”三个条件。
2. 对外提供能力的任务，必须至少有一种可复核证据：
   - 自动化测试
   - 手工验收记录
   - 接口返回样例
   - 页面截图
   - 工件文件
   - 日志或状态面输出
3. 涉及基础设施、网关、监控、调度、消息链路的任务，必须提供运行态证据，不能只提交配置文件。
4. 涉及真实外部系统的任务，在外部环境不可得时，可先验：
   - 本地实现
   - 配置模板
   - mock 或 local-real 验收
   - 联调清单
   - 阻塞说明
5. 涉及安全、权限、审计、成本控制的任务，必须补齐规则说明和回归验证。
6. 涉及业务闭环的任务，必须关注从“输入 -> 处理 -> 输出 -> 回流”的完整链路，而不只是单点接口。

## 4. 通用证据要求

| 证据类型 | 说明 |
| --- | --- |
| 代码实现 | 关键模块、接口、脚本、配置文件已提交 |
| 自动化测试 | `pytest`、接口回归、脚本验收、状态检查 |
| 手工验收记录 | curl、Postman、页面操作、终端输出 |
| 工件文件 | `artifacts/`、`docs/`、导出文件、manifest、dashboard、告警规则 |
| 运行态证据 | 健康检查、状态面、监控图、日志、容器状态、服务端点返回 |
| 阻塞说明 | 外部环境、凭证、GPU、K8s、IdP 等不可得时的正式记录 |

## 8. 当前执行建议

基于 `任务清单.md` 当前状态，当前验收应按以下顺序推进：

1. 先收口 `P0`、`P2`、`P3`、`P4`、`P6`、`P9`、`P12` 这些已大面积完成的域。
2. 再重点处理 `P1`、`P5`、`P7`、`P8`、`P10` 中“本地已完成、真实环境未完成”的任务。

一句话口径：

**当前项目验收应以“本地真实可运行 + 关键链路可复核 + 外部阻塞有证据”作为主标准，而不是以“所有目标态基础设施都已上线”作为前置条件。**
